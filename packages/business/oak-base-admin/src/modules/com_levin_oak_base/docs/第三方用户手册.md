# Levin Oak 基础模块补充手册

本文是 `@levin/oak-base-admin` 基础业务模块的补充说明。当前前端项目整体定位是基础组件和模块化后台工作区，第三方集成的主手册见 `@levin/admin-framework` 包内的 `src/framework-commons/docs/第三方用户手册.md`。本文件只说明 Oak 基础模块的安装、启用、页面和后端契约。本模块 npm 包的正式运行入口是构建后的 `dist`；随包携带的 `src` 只用于源码查看、调试和问题定位，不作为第三方应用的公共编译入口。

## 1. 模块定位

`@levin/oak-base-admin` 是 Levin Oak 基础后台业务模块包，模块名为 `com.levin.oak.base`，默认后端 API 前缀为 `/com.levin.oak.base/V1/api`。它依赖 `@levin/admin-framework` 提供应用壳、公共组件和运行时能力，本包只提供基础业务页面、API service、页面注册表、后端菜单路由映射和国际化资源。

第三方项目通常需要同时安装并启用：

- `@levin/admin-framework`：后台运行框架、页面注册表、运行时请求代理、布局、权限、CRUD 公共能力。
- `@levin/oak-base-admin`：基础模块业务页面和业务 API。

模块适合以下接入方式：

- 最终后台应用通过 npm 私服依赖安装业务模块。
- 第三方应用通过包根入口或文档声明的子路径使用 `dist` 构建结果，不直接从 `@levin/oak-base-admin/src/...` 引入源码。
- 后端继续提供菜单、权限和业务接口，前端模块只提供已注册页面和 API 调用能力。
- 项目需要覆盖少量页面时，在最终应用的 `src/pages` 下放同名页面覆盖文件，不复制整个模块源码。

## 2. 安装要求

运行环境：

- Node.js `>=20.19.0`
- pnpm `>=10.0.0`
- Vue `>=3.5.0`
- Vue Router `>=4.0.0`
- Pinia `>=3.0.0`
- Ant Design Vue `>=4.0.0`

安装示例：

```bash
pnpm add @levin/admin-framework@5.6.8 @levin/oak-base-admin@5.6.8
```

依赖应从 npm 私服安装。交付或生产项目不建议复制 `packages/business/oak-base-admin/src` 源码作为长期集成方式，也不要从 `@levin/oak-base-admin/src/...` 深路径导入源码。

发布包内可能包含 `src` 目录，它是第三方开发过程中的辅助资料。正式应用编译应以 `package.json` 的 `main`、`module`、`types` 和 `exports` 中声明的 `dist` 入口为准。

## 3. 应用启用

### 3.1 注册模块

最终应用应在模块列表中启用基础模块：

```ts
import type { AdminFrontendModule } from '@levin/admin-framework';

import { createOakBaseAdminModule } from '@levin/oak-base-admin';

export const enabledFrontendModules: AdminFrontendModule[] = [
  createOakBaseAdminModule(),
];
```

如需关闭默认 CRUD 路由，只保留页面注册表和后端菜单映射：

```ts
createOakBaseAdminModule({ crud: false });
```

### 3.2 配置后台运行时

应用挂载前必须配置运行时服务。基础模块页面和 API 通过这些服务访问后端、同步菜单、读取消息和处理用户安全能力。

```ts
import {
  configureAdminApplication,
  normalizeAdminGlobPageMap,
  type AdminPageMap,
} from '@levin/admin-framework';
import {
  menuService,
  noticeService,
  userService,
} from '@levin/oak-base-admin/modules/com_levin_oak_base/api/index';

import { enabledFrontendModules } from './modules/list';

const pageOverrides = normalizeAdminGlobPageMap(
  import.meta.glob('./pages/**/*.vue') as AdminPageMap,
  './pages',
);

configureAdminApplication({
  menuSyncService: menuService,
  modules: enabledFrontendModules,
  noticeService,
  pageOverrides,
  userSecurityService: userService,
});
```

如果项目直接使用 `setAdminFrameworkRuntime`，必须至少提供 `requestClient`，并按需提供 `menuSyncService`、`noticeService` 和 `userSecurityService`。

### 3.3 页面加载和菜单来源

模块提供隐藏路由、页面映射和后端路由映射。真实菜单应以后端当前用户授权菜单为准，前端根据菜单 `path` 匹配已注册页面。

超级管理员执行“上传页面路由”时，入口应用会上传所有已启用前端模块的页面路由，不只上传基础模块或当前打开的模块。基础模块的上传项会使用本模块自己的模块 ID：`com.levin.oak.base`。

基础模块的 `backendRouteMappings` 必须为每条页面映射同步维护：

- `path`：后端菜单或控制器路由路径，也是上传 payload 中的菜单入口字段。
- `viewPath`：前端运行时页面注册路径，例如 `/system/com_levin_oak_base/user/index.vue`。
- `sourceFilePath`：相对于前端源码目录的页面文件路径，例如 `modules/com_levin_oak_base/views/user/index.vue`。

这几个字段的职责不要混用：`path` 是后端菜单入口；`viewPath` 是运行时真正用于加载页面组件的注册路径；`sourceFilePath` 仅用于展示、排查和源码定位。历史文档中出现过的 `sourcePath`、`targetPath`、`view` 和 `sourceFile` 已废弃，第三方模块不要再使用。

这样拆分的目的是把“后端菜单入口”“前端运行时页面”和“源码调试位置”解耦：后端只关心菜单和权限挂在哪个 `path` 下；前端运行时只根据 `viewPath` 找到已经注册的页面组件；调试和生成工具再通过 `sourceFilePath` 追溯页面来源。第三方模块接入时不需要猜测页面来自哪个包，也不需要把源码路径当作运行时路径使用，模块覆盖和最终应用自定义页面也能保持稳定。

`moduleId` 不作为后端接口必填项；但模块存在自己的模块 ID 时，上传时每个菜单项都必须带上该菜单所属模块的模块 ID。

上传页面路由弹窗默认全选全部本地页面路由，允许在上传前按树节点取消或重新选择。选择状态按树形菜单语义处理：选择父节点会同步选择其全部子节点；取消父节点会同步取消其全部子节点；只选择部分叶子节点时，上级节点显示为半选状态，上传时只提交已选择的节点及维持层级所需的父节点。

默认全选的目的是保证常规发布时同步完整菜单，避免第三方模块新增页面后遗漏上传；按树选择的目的则是给调试、灰度和清理历史菜单留出人工控制空间。即使只选择叶子节点，上传 payload 也会保留必要父节点层级，确保后端菜单树结构完整，但不会把未选择的兄弟节点误同步上去。

默认 CRUD 路由根路径为：

```text
/admin-crud
```

后端菜单常用源路径为：

```text
/clob/V1/<Resource>
```

页面注册表路径为：

```text
/system/com_levin_oak_base/<page>/index.vue
```

## 4. 接口和代理配置

基础模块 API 前缀：

```text
/com.levin.oak.base/V1/api
```

开发服务器和生产 Nginx 需要正确代理以下路径：

- `/com.levin.oak.base`：基础模块后端 API。
- `/lfs`：文件、图片和附件资源。
- `/admin`：后端管理或直出页面。
- `/api-docs`、`/swagger`：接口文档。
- `/.well-known/acme-challenge`：证书校验路径。

前端未命中的页面路由不应直接返回前端 404，应先回退给后端处理；只有前端和后端都无法处理时才返回 404。

## 6. 常用业务能力

### 6.1 通用 CRUD 页面

大多数页面基于模块内的显式 `config.ts` 配置，支持：

- 查询、分页、排序和刷新。
- 新增、编辑、详情、删除、批量删除等常规操作。
- 表格列设置、全屏、导出。
- 表单字段按业务类型选择组件，例如租户选择、组织树、角色选择、图片上传、文件上传、JSON 编辑、代码编辑、开关、日期时间、标签输入等。
- 权限控制来自后端控制器 `@ResAuthorize`、`@CRUD.Op` 等元数据，前端按钮会按权限和操作归属显示。

本模块内置的通用 CRUD 页面沿用 `@levin/admin-framework` 的统一查询表单布局：查询面板使用响应式等宽列网格，列数按查询面板容器宽度计算，不按整个浏览器视口计算；默认每列最小可用宽度约为 `280px`，列数最少 1 列、最多 7 列。查询字段和查询、重置、更多/收起按钮位于同一套网格中，操作区靠右显示，当前行没有可用列时自然换到下一行。第三方项目覆盖这些页面或新增同类 CRUD 页面时，应保持同一布局规则，避免按菜单路径、页面路径或字段数量写局部特判。

### 6.2 用户、角色和权限

`用户管理`、`角色管理`、`权限项定义`、`组织管理`、`菜单管理` 是 RBAC 相关核心页面。用户和角色页面支持角色选择、组织归属、启用状态、权限配置和数据权限相关操作。菜单页面支持菜单基础信息、页面路径、操作按钮配置和后端菜单同步。

### 6.3 租户和站点

`租户管理`、`租户应用`、`租户站点` 用于维护多租户基础数据。租户和站点配置包含 Logo、图标、版权、技术支持、前端展示扩展信息 `uiExInfo` 等字段。租户站点返回的 `uiExInfo.admin-ui-base-setting` 会影响后台基础界面配置。

### 6.4 通知和我的消息

`通知公告` 管理通知内容，`通知处理日志` 查看处理记录。`我的消息` 页面会调用 `noticeService.myMessages` 加载当前用户消息，支持未读、已读、全部筛选，查看详情，单条或批量标记已读。

### 6.5 系统设置和租户系统设置

`系统设置` 维护配置项定义。`租户系统设置` 面向当前租户编辑启用的配置项，支持文本、数字、开关、下拉、JSON、代码编辑和 JsonSchema 动态表单。配置项可按分类和分组展示，保存时只提交发生变化且可编辑的配置。

### 6.6 文件、内容和支付

模块内置文件资源、文章、栏目、品牌、地址、区域、国家地区、资金账户、支付渠道、支付订单等基础业务页面。图片和文件字段通过框架统一上传能力处理，后端资源路径需要代理 `/lfs`。

## 7. 页面覆盖和二次开发

最终应用需要改造某个默认页面时，应通过页面覆盖机制处理。不要复制整个模块目录。

覆盖优先级：

```text
最终应用 src/pages 覆盖层
  > 最终应用内置页面
  > 已启用业务模块 pageMap
  > admin-framework 公共 pageMap
```

示例：覆盖角色管理页面。

```text
业务模块默认页面：
src/modules/com_levin_oak_base/views/role/index.vue

最终应用覆盖文件：
src/pages/system/com_levin_oak_base/role/index.vue
```

覆盖文件只放项目差异化内容，通用默认页面继续由 npm 包提供。

## 8. Tailwind 和样式集成

最终应用的 Tailwind 扫描范围必须包含安装后的 `@levin`、`@vben` 和 `@vben-core` 包，尤其是 pnpm 的 `.pnpm` 传递依赖路径。否则菜单、页签、布局、按钮和业务页面中的 Tailwind 工具类可能不会生成。

`postcss.config.mjs` 应显式导入本应用的 `tailwind.config.mjs`，并配置 `tailwindcss: { config: tailwindConfig }`，保证依赖包里的样式类被正确处理。

业务页面样式应跟随框架主题变量。第三方项目覆盖页面时，优先使用语义类，例如 `bg-background`、`text-foreground`、`border-border`、`text-primary`，避免写死颜色值。

## 9. 权限、菜单和后端契约

第三方项目接入时需要后端满足以下契约：

- 当前登录用户接口可提供用户身份和权限判断所需信息。
- 菜单接口返回当前用户可见菜单，菜单 `path` 能匹配模块上传的菜单 `path` 或页面注册路径。
- 基础模块 API 前缀 `/com.levin.oak.base/V1/api` 下暴露对应控制器接口。
- 文件上传和回显资源路径可访问。
- `tenantSiteInfo` 或租户站点接口返回的界面配置能被前端正常读取。

前端不要硬编码角色名、用户名或权限码来推断管理员身份；管理员能力应以后端当前用户信息和权限结果为准。

## 10. 发布和升级建议

模块构建结果发布到 npm 私服后，第三方项目通过版本号升级：

```bash
pnpm add @levin/oak-base-admin@5.6.8
```

建议升级步骤：

1. 修改最终应用依赖版本。
2. 执行 `pnpm install`。
3. 启动开发服务，检查登录、菜单加载、页面打开、关键接口和控制台错误。
4. 构建最终应用。
5. 在测试环境验证上传资源、权限按钮、租户站点 UI 配置和常用 CRUD 页面。

如果需要本地联调模块源码，可以使用 workspace 或 `link:` 方式连接模块仓库；测试、生产和交付环境应使用 npm 私服包，并通过包根入口或已声明的 `dist` 子路径导出引入模块。

## 11. 常见问题

### 页面菜单存在但打开空白

检查菜单 `path` 是否能匹配模块注册路径，检查最终应用是否已启用 `createOakBaseAdminModule()`，并确认页面覆盖文件没有路径拼写错误。

### 接口 404

确认代理是否覆盖 `/com.levin.oak.base`，并确认后端模块接口路径为 `/com.levin.oak.base/V1/api`。

### 图片或附件无法回显

确认 `/lfs` 等资源路径已转发到后端，生产 Nginx 和开发代理需要保持一致。

### 样式缺失或菜单显示异常

确认 Tailwind 扫描范围包含 `node_modules/@levin`、`node_modules/@vben`、`node_modules/@vben-core` 以及 pnpm `.pnpm` 下的传递依赖路径。

### 发布同版本失败

npm 包版本不可重复发布。业务模块发版前应确认私服中已有版本，并在 `package-versions.json` 中为对应包配置新版本后同步。
