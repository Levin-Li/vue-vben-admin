# Levin 后台基础组件项目第三方用户手册

本文面向使用本项目 npm 私服包的第三方团队。当前项目定位是后台基础组件和模块化前端工作区，不是单一业务系统源码。第三方项目应通过 npm 包安装、装配和覆盖来使用本项目能力，避免长期复制框架或基础模块源码。npm 包的正式运行入口是构建后的 `dist`；随包携带的 `src` 只用于源码查看、调试和问题定位，不作为第三方应用的公共编译入口。

## 1. 项目定位

本项目提供后台系统的基础组件、公共运行时、模块化装配能力和基础业务模块。推荐集成形态：

```text
最终后台应用
  = @levin/admin-framework
  + @levin/oak-base-admin
  + 一个或多个第三方业务模块包
  + 薄宿主应用 bootstrap-app
```

核心原则：

- 公共能力从 npm 私服安装，不复制源码。
- 第三方应用通过包根入口或文档声明的子路径使用 `dist` 构建结果，不直接从 `@levin/*/src/...` 引入源码。
- 最终应用只负责启动、后端地址、主题、偏好、启用模块和页面覆盖。
- 业务页面和 API 放在各自模块包中独立发布。
- 菜单、权限和页面可见性以后端当前用户授权结果为准。

## 2. 包分层

### 2.1 公共框架包

`@levin/admin-framework` 是基础组件项目的核心入口，提供：

- 应用启动和挂载能力。
- 登录、布局、路由守卫、菜单同步和 RBAC 权限判断。
- 通用 CRUD 页面、表格、表单、弹窗、上传、JSON/代码编辑、权限组件。
- 页面注册、页面覆盖、模块注册、国际化合并。
- 请求适配、API service 基类、事件总线、顶部栏扩展区、窗口级可拖拽悬浮面板。

第三方项目优先从稳定入口导入：

```ts
import { configureAdminApplication } from '@levin/admin-framework';
import { bootstrap } from '@levin/admin-framework/framework-commons/app/bootstrap';
```

### 2.2 基础业务模块包

`@levin/oak-base-admin` 提供 `com.levin.oak.base` 基础模块，包含用户、角色、权限、菜单、组织、租户、通知、文件资源、系统设置等后台基础页面和 API。

基础模块默认后端 API 前缀：

```text
/com.levin.oak.base/V1/api
```

### 2.3 Vben 基础组件包

项目同时发布一组 `@vben` 和 `@vben-core` 基础能力包，例如：

- `@vben/common-ui`：常用 UI 容器和业务 UI。
- `@vben/layouts`：后台布局、菜单、页签。
- `@vben/request`：请求能力。
- `@vben/preferences`：偏好设置。
- `@vben/stores`：通用状态。
- `@vben/icons`：图标能力。
- `@vben-core/form-ui`、`@vben-core/menu-ui`、`@vben-core/tabs-ui`、`@vben-core/popup-ui`：更底层的 UI 能力。

普通第三方应用通常不需要直接安装所有包，只需安装 `@levin/admin-framework`、业务模块包和 peerDependencies 要求的运行时依赖。

### 2.4 常用公共能力速查

第三方项目优先使用下列公共入口，不要复制框架源码或从包内 `src` 深路径引入：

| 能力 | 推荐入口 | 说明 |

| 应用启动 | `@levin/admin-framework/framework-commons/app/bootstrap` | 宿主应用启动入口。 |
| 公共 API service | `@levin/admin-framework/framework-commons/app/api` | 文件上传、RBAC、OAuth、枚举、缓存等公共服务统一导出。 |
| 文件和图片上传 | `@levin/admin-framework/framework-commons/app/api/file-storage-service` | `fileStorageService`、上传路径常量、`uploadFileByFileStorageController`。 |

### 2.5 公共 API service 清单

框架公共 API service 统一从 `@levin/admin-framework/framework-commons/app/api` 导出。第三方模块需要上传、权限、登录、枚举、缓存、JsonSchema 或插件管理等公共能力时，应优先使用这些 service，不要在页面组件里直接拼接公共接口地址。

可以按聚合入口导入：

```ts
import {
  fileStorageService,
  rbacService,
} from '@levin/admin-framework/framework-commons/app/api';
```

```ts
import { fileStorageService } from '@levin/admin-framework/framework-commons/app/api/file-storage-service';
```

## 3. 安装

推荐使用 pnpm：

```bash
pnpm add @levin/admin-framework@<version> @levin/oak-base-admin@<version>
```

具体版本以私服中已发布版本和业务模块 `peerDependencies` 为准。升级公共框架包时，应同步升级依赖它的业务模块包版本约束，避免最终应用安装到不匹配的框架和业务模块组合。

运行环境要求：

- Node.js `>=20.19.0`
- pnpm `>=10.0.0`
- Vue `>=3.5.0`
- Vue Router `>=4.0.0`
- Pinia `>=3.0.0`
- Ant Design Vue `>=4.0.0`

依赖应从 npm 私服安装。测试、生产和交付环境不要使用复制源码的方式集成公共框架或基础模块，也不要从 `@levin/admin-framework/src/...` 或 `@levin/*-admin/src/...` 深路径导入源码。

发布包内可能包含 `src` 目录，它是第三方开发过程中的辅助资料：用于阅读实现、排查问题、提交问题定位信息或本地临时联调。正式应用编译应以 `package.json` 的 `main`、`module`、`types` 和 `exports` 中声明的 `dist` 入口为准。

## 4. 薄宿主应用接入

第三方最终应用应保持“薄宿主”结构，只做装配：

```text
src/
  main.ts
  modules/list.ts
  pages/                # 页面覆盖层
  preferences.ts
```

### 4.1 声明启用模块

```ts
import type { AdminFrontendModule } from '@levin/admin-framework';

import { createOakBaseAdminModule } from '@levin/oak-base-admin';

export const enabledFrontendModules: AdminFrontendModule[] = [
  createOakBaseAdminModule(),
];
```

第三方业务模块也应导出类似的模块工厂函数，然后加入 `enabledFrontendModules`。

### 4.2 配置应用服务

```ts
import {
  configureAdminApplication,
  normalizeAdminGlobPageMap,
  type AdminPageMap,
} from '@levin/admin-framework';
import {
  menuService,
  noticeService,
  userService,
} from '@levin/oak-base-admin/modules/com_levin_oak_base/api/index';

import { enabledFrontendModules } from './modules/list';

const pageOverrides = normalizeAdminGlobPageMap(
  import.meta.glob('./pages/**/*.vue') as AdminPageMap,
  './pages',
);

configureAdminApplication({
  menuSyncService: menuService,
  modules: enabledFrontendModules,
  noticeService,
  pageOverrides,
  userSecurityService: userService,
});
```

### 4.3 启动应用

```ts
const { bootstrap } =
  await import('@levin/admin-framework/framework-commons/app/bootstrap');

await bootstrap(namespace);
```

`namespace` 建议由项目标识、版本和环境组成，用于隔离偏好设置和本地存储。

## 5. 页面、菜单和权限

模块提供页面注册表和后端菜单路径映射。真实菜单应以后端当前用户授权菜单为准，前端根据菜单路径匹配模块注册页面。

超级管理员执行“上传页面路由”时，必须上传最终应用中所有已启用前端模块的页面路由，不得只上传当前模块、默认模块或基础模块。每个业务模块应维护自己的 `backendRouteMappings`，并在每条页面映射中提供：

- `path`：后端菜单或控制器路由路径，也是上传 payload 中的菜单入口字段。
- `viewPath`：前端运行时页面注册路径，例如 `/system/com_levin_oak_base/user/index.vue`。
- `sourceFilePath`：相对于前端源码目录的页面文件路径，例如 `modules/com_levin_oak_base/views/user/index.vue`。

这几个字段的职责不要混用：`path` 是后端菜单入口；`viewPath` 是运行时真正用于加载页面组件的注册路径；`sourceFilePath` 仅用于展示、排查和源码定位。历史文档中出现过的 `sourcePath`、`targetPath`、`view` 和 `sourceFile` 已废弃，第三方模块不要再使用。

这样拆分的目的是把“后端菜单入口”“前端运行时页面”和“源码调试位置”解耦：后端只关心菜单和权限挂在哪个 `path` 下；前端运行时只根据 `viewPath` 找到已经注册的页面组件；调试和生成工具再通过 `sourceFilePath` 追溯页面来源。第三方模块接入时不需要猜测页面来自哪个包，也不需要把源码路径当作运行时路径使用，模块覆盖和最终应用自定义页面也能保持稳定。

`moduleId` 不作为后端接口必填项；但前端模块存在自己的模块 ID 时，上传时每个菜单项都必须带上该菜单所属模块的模块 ID，通常取 `AdminFrontendModule.name`。

上传页面路由弹窗默认全选全部本地页面路由，允许在上传前按树节点取消或重新选择。选择状态按树形菜单语义处理：选择父节点会同步选择其全部子节点；取消父节点会同步取消其全部子节点；只选择部分叶子节点时，上级节点显示为半选状态，上传时只提交已选择的节点及维持层级所需的父节点。

默认全选的目的是保证常规发布时同步完整菜单，避免第三方模块新增页面后遗漏上传；按树选择的目的则是给调试、灰度和清理历史菜单留出人工控制空间。即使只选择叶子节点，上传 payload 也会保留必要父节点层级，确保后端菜单树结构完整，但不会把未选择的兄弟节点误同步上去。

页面解析优先级：

```text
最终应用 src/pages 覆盖层
  > 最终应用内置页面
  > 已启用业务模块 pageMap
  > admin-framework 公共 pageMap
```

覆盖角色管理页面示例。这里的包内源码路径只用于说明默认页面来源，不是第三方应用的 import 路径：

```text
业务模块默认页面源码参考：
@levin/oak-base-admin/src/modules/com_levin_oak_base/views/role/index.vue

最终应用覆盖文件：
src/pages/system/com_levin_oak_base/role/index.vue
```

覆盖文件只放项目差异化内容，不要复制整个业务模块目录。

权限按钮、菜单可见性、数据权限等应以后端返回和框架 RBAC 判断为准。前端不要硬编码角色名、用户名或权限码来推断管理员身份。

## 6. 基础组件和公共能力

### 6.1 通用 CRUD

框架提供通用 CRUD 页面能力，模块页面通过显式配置接入，支持：

- 查询、分页、排序、刷新。
- 新增、编辑、详情、删除、批量删除。
- 表格列设置、全屏、导出。
- 权限控制和按钮显隐。
- 多类型表单组件，包括选择、树选择、上传、JSON、代码编辑、开关、日期时间、标签输入等。

查询表单默认使用响应式等宽列网格。列数按查询面板自身容器宽度计算，而不是按整个浏览器视口计算；默认每列最小可用宽度约为 `280px`，列数最少 1 列、最多 7 列。字段和操作按钮参与同一套网格布局：查询、重置、更多/收起按钮位于操作区内并靠右；当前行没有可用列时，操作区自然换到下一行。简单模式收起时，`searchCollapsedCount` 只是基础展示数量，框架会继续追加后续查询项，尽量填满操作区所在行前面的完整空列；因此无论操作区在第一行还是换到下一行，它前面的可用列都会优先展示查询项。高级查询展开后仍按同一规则排布。第三方页面通常只需要声明查询字段，不需要为按钮区单独写布局。

如果页面有特殊筛选器、跨列字段、顶部全局筛选条或其它明确交互需求，应通过页面配置显式表达，不要按字段名、菜单路径或页面路径硬编码特判。

### 6.2 事件总线

跨模块通知使用 `@levin/admin-framework/framework-commons` 导出的事件总线。API 请求事件统一使用：

```text
type: api.request
topic: 本次请求地址
data: 完整请求结果包
```

事件总线不替代 API service、Pinia、路由或组件 props。

### 6.3 顶部栏扩展区

需要向顶部栏中心或右侧注册控件时使用：

```ts
import { useLayoutHeaderExtensionArea } from '@levin/admin-framework/framework-commons';

const topRight = useLayoutHeaderExtensionArea('right');
topRight.add({
  id: 'project-current-context',
  component: CurrentContextSelect,
  order: 20,
});
```

### 6.4 窗口级可拖拽悬浮面板

需要跨页面内容区浮动的上下文切换器、筛选条或工具面板时使用：

```ts
import { useDraggableFloatingPanels } from '@levin/admin-framework/framework-commons';

const panels = useDraggableFloatingPanels('project-context');
panels.add({
  id: 'tenant-filter',
  component: TenantFilter,
  name: '当前租户筛选',
  panelProps: {
    storageKey: 'project-context:tenant-filter',
  },
});
```

普通业务页面不要自行挂载全局 host，也不要直接引用框架内部实现文件。

### 6.5 用户下拉菜单扩展区

需要在用户头像下拉菜单中添加第三方动作按钮时使用：

```ts
import { useUserDropdownMenuItems } from '@levin/admin-framework/framework-commons';

const userDropdownMenus = useUserDropdownMenuItems();
userDropdownMenus.add({
  id: 'project-help',
  icon: 'lucide:circle-help',
  order: 500,
  text: '项目帮助',
  handler: () => {
    window.open('/project/help', '_blank');
  },
});
```

第三方按钮只会出现在公共模块固定动作和 `锁定屏幕` 之间。`个人中心`、`上传页面路由`、`上传界面设置`、`监听器管理`、`锁定屏幕`、`退出登录` 都是公共固定区域，第三方模块不能替换或删除。

需要主动移除时使用注册项 `id`：

```ts
import { removeUserDropdownMenuItem } from '@levin/admin-framework/framework-commons';

removeUserDropdownMenuItem('project-help');
```

`useUserDropdownMenuItems().add(...)` 注册的菜单会在当前组件卸载时自动移除；入口应用注册的常驻菜单应保存清理函数，或在不再需要时调用 `removeUserDropdownMenuItem(id)`。

### 6.6 文件和图片上传

文件和图片上传统一通过基础模块后端 `FileStorageController`，前端公共封装在：

```ts
import {
  FILE_STORAGE_MULTI_UPLOAD_PATH,
  FILE_STORAGE_SINGLE_UPLOAD_PATH,
  fileStorageService,
  uploadFileByFileStorageController,
} from '@levin/admin-framework/framework-commons/app/api/file-storage-service';
```

常用场景：

```ts
const url = await uploadFileByFileStorageController(file);
```

如果当前业务模块需要带自己的 API 模块前缀，可传入 `moduleBase`：

```ts
const url = await uploadFileByFileStorageController(
  file,
  '/com.levin.oak.base/V1/api',
);
```

后端单文件上传接口返回 `ApiResp.ok(url)`，前端公共封装会把结果归一化为字符串 URL；如果接口没有返回可用 URL，会抛出“上传接口未返回文件地址”。多文件上传可使用 `fileStorageService.uploadFiles(file)` 或 `FILE_STORAGE_MULTI_UPLOAD_PATH` 配合业务上传组件。

上传成功后，页面应立即把返回的 URL 写回表单字段，并同步上传组件的 fileList。图片或附件回显依赖后端资源路径代理，宿主应用必须正确转发 `/lfs` 等资源路径。

## 7. 样式和 Tailwind

第三方宿主应用必须扫描安装后的组件包源码，否则依赖包中的 Tailwind 类可能不会生成。

```js
content: [
  './index.html',
  './src/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/@levin/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/@vben/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/@vben-core/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/.pnpm/@levin+*/node_modules/@levin/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/.pnpm/@vben+*/node_modules/@vben/**/*.{vue,js,ts,jsx,tsx,html}',
  './node_modules/.pnpm/@vben-core+*/node_modules/@vben-core/**/*.{vue,js,ts,jsx,tsx,html}',
];
```

`postcss.config.mjs` 应显式导入本应用的 `tailwind.config.mjs`，并配置 `tailwindcss: { config: tailwindConfig }`。

覆盖页面和自定义组件应使用框架语义色和 Tailwind 语义类，例如 `bg-background`、`text-foreground`、`border-border`、`text-primary`，避免写死颜色值。

## 8. 后端代理要求

开发代理和生产 Nginx 至少应覆盖：

- `/com.levin.oak.base`：基础模块 API。
- 第三方业务模块自己的后端模块前缀。
- `/lfs`：文件、图片和附件资源。
- `/admin`：后端管理或直出页面。
- `/api-docs`、`/swagger`：接口文档。
- `/.well-known/acme-challenge`：证书校验路径。

前端未命中的页面路由应先回退给后端处理；只有前端和后端都无法处理时才返回 404。

## 9. 第三方业务模块开发约定

业务模块包建议结构：

## 10. 发布和升级

本项目使用 npm 私服发版。发布时必须显式使用可写 hosted registry 和认证，不要把 token 写进仓库。发布物的正式入口必须是 `dist` 构建结果；`src` 可以随包携带为辅助资料，但不得作为公开 `exports` 暴露给第三方应用编译。

```bash
NPM_REGISTRY=http://nexus.v-ma.com/repository/npm/ \
NPM_AUTH_FROM_MAVEN=true \
MAVEN_SERVER_ID=dist-repo \
pnpm run publish:packages -- --only=@levin/admin-framework
```

业务模块发布示例：

```bash
NPM_REGISTRY=http://nexus.v-ma.com/repository/npm/ \
NPM_AUTH_FROM_MAVEN=true \
MAVEN_SERVER_ID=dist-repo \
pnpm run publish:admin-modules -- --only=@levin/oak-base-admin
```

安装依赖时通常使用 group 仓库，例如：

```text
http://nexus.v-ma.com/repository/npm-public/
```

发布同版本会失败。发版前应确认私服中已有版本，并在 `package-versions.json` 中配置新版本后同步。

发布前还应确认：

- `main`、`module`、`types` 和默认 `exports` 均指向 `dist`。
- `exports` 中没有 `./src/*` 或其他指向 `src` 的公共导出。
- 需要给第三方查看的源码和文档可以继续通过 `files` 随包携带。

## 11. 常见问题

### 编译或打包阶段看到 warning

本项目定位是前端公共模块集合，发布包主要面向 Vue/Vite 后台应用消费，不是纯 Node.js 运行库。第三方项目在编译或打包阶段可能看到一些来自公共模块、SFC、动态导入或样式处理的提示；只要最终构建命令退出码为 0，页面可打开，核心功能可用，这类提示通常不是致命问题。

常见非致命提示包括：

- `vue-sfc-transformer is not installed`：表示公共包构建时没有把所有 `.vue` SFC 完全转成普通 JS。当前包允许发布 `.vue` 文件，由最终应用的 Vite/Vue 插件处理。
- `dynamic import will not move module into another chunk`：表示同一个模块同时被静态导入和动态导入，影响的是分包优化，不代表模块不可用。
- 打包日志中出现较大的 chunk 或 CSS 文件：通常是通用后台框架、VXE Table、JSON 编辑器、表单组件等公共能力集中打包导致。需要优化首屏性能时再按实际页面拆分，不应先按日志大小盲目删除公共能力。

需要处理的阻断问题通常具备以下特征：

- 构建命令退出码非 0。
- Vite、Sass、TypeScript 或 Rollup 明确提示找不到文件、找不到导出、语法无法解析。
- 页面运行时出现白屏、路由打不开、核心 API 调用失败或控制台新增错误。
- 发布包的 `exports` 指向不存在的 `dist` 文件。

遇到提示时，建议先按顺序确认：最终构建是否成功、应用页面是否能打开、提示是否来自公共模块的非阻断构建策略。如果构建成功但提示影响第三方判断，可以把完整日志提交给公共模块维护方评估是否需要收敛日志或优化打包策略。

### 页面打开空白

检查是否启用了对应模块、后端菜单路径是否能匹配页面注册表、最终应用覆盖页面路径是否正确。

### 样式缺失或菜单页签异常

检查 Tailwind `content` 是否扫描了 `node_modules/@levin`、`@vben`、`@vben-core` 以及 pnpm `.pnpm` 下的传递依赖。

### 接口 404

检查代理是否覆盖对应后端模块前缀，基础模块为 `/com.levin.oak.base`。

### 图片或附件无法回显

检查 `/lfs` 等资源路径是否转发到后端。

### 图片或文件上传失败

先确认页面使用的是 `@levin/admin-framework/framework-commons/app/api/file-storage-service` 的公共上传封装，上传请求是否命中后端 `FileStorageController`。单文件上传应返回字符串 URL；如果返回对象或空值，公共封装无法稳定回写表单，应优先修正后端返回。

### 多模块页面冲突

确认每个模块的 `name`、路由 `name`、页面注册路径和后端菜单 `path` 都带有稳定模块语义，避免不同模块使用同一标识。
