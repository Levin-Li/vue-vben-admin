## 管理后台页面开发总原则

前端项目页面的开发要基于后端控制器类和注解元数据，不是只基于 OpenAPI 文档，也不是前端自行猜测页面结构。

所有前端页面开发规则，都是用来指导页面开发的，不是用来构建页面代码生成器的。

不允许把这些规则实现成页面代码生成器、运行时元数据生成器或通用页面自动拼装系统。

不允许用控制器元数据、OpenAPI 元数据或其他中间结构替代真实页面开发。

规则的作用是指导开发者如何分析后端源码、如何设计页面结构、如何确定字段、按钮、权限、布局和组件，而不是生成页面代码本身。

每个菜单页面仍应有明确的独立页面目录和页面实现。

每个菜单对应一个明确的前端页面目录。

每个菜单页面应有自己的页面入口文件，例如 `index.vue`。

页面之间可以复用公共组件，但不能把多个菜单页面混成一个“通用生成页”。

共享组件只负责复用局部能力，不负责替代页面本身。

## 管理后台页面开发补充规则

### 一、页面开发主依据

1. 在本项目中，只要存在对应的 Spring MVC 控制器源码，就不得使用 OpenAPI 作为页面开发主依据。

2. OpenAPI 只能作为辅助信息，用于以下场景：
   - 辅助确认某个控制器是否在当前运行环境实际暴露。
   - 辅助确认某个控制器方法是否实际暴露。
   - 辅助排查接口路径、HTTP 方法、运行时条件注解是否生效。
   - 辅助排查源码与运行时暴露不一致的原因。

3. OpenAPI 不能用于决定以下内容：
   - 页面是否存在。
   - 页面字段。
   - 页面按钮。
   - 菜单归属。
   - 权限点。
   - 显隐条件。
   - 表单布局。
   - 组件类型。
   - 业务交互行为。

4. 页面开发的主依据必须是后端源码：
   - 业务控制器。
   - 父类控制器。
   - 控制器方法参数。
   - 控制器方法返回值。
   - 方法参数对应的 DTO / BO / VO / 基本类型参数。
   - 返回对象对应的 DTO / Info / VO / BO。
   - 实体类及其关联类型。
   - 相关注解，如 `@CRUD`、`@CRUD.ListTable`、`@CRUD.Op`、`@DisableApiOperation`、`@Operation`、`@Tag`、`@Schema`、`@Options`、Java Validation 注解等。

### 二、有效控制器判定

1. 只有有效的 Spring MVC 控制器才对应前端页面模块。

2. 一个业务控制器要被视为有效页面控制器，必须同时满足：
   - 是 Spring Bean。
   - 不是抽象类。
   - 是 Spring MVC 控制器，例如存在 `@RestController`、`@Controller`，或通过其他 Spring 配置明确注册为 MVC 控制器。
   - 自身或父类存在 `@CRUD` 注解。

3. 如果业务控制器自身没有 `@CRUD`，但父类控制器有 `@CRUD`，则可以继承父类的 CRUD 页面定义。

4. 如果业务控制器是抽象类，即使源码里有 `@RequestMapping`、`@CRUD` 或继承了 CRUD 控制器，也不能直接作为前端页面控制器。

5. 如果业务控制器没有作为 Spring Bean 注册，例如缺少控制器注解、没有被组件扫描、被条件注解禁用，不能直接作为页面控制器。

6. `@ConditionalOnProperty` 等条件注解需要纳入判断。
   - 如果当前配置明确禁用了该控制器，则对应页面不应展示。
   - 如果无法仅凭源码判断条件是否生效，可以结合运行时信息或 OpenAPI 辅助确认。

7. 如果不确定控制器是否有效，可以结合 OpenAPI 辅助判断。
   - OpenAPI 中存在对应接口，说明当前运行环境大概率暴露了该控制器或方法。
   - OpenAPI 中不存在对应接口，应继续检查是否由 `@DisableApiOperation`、条件注解、抽象类、非 Spring Bean、运行环境配置、扫描范围等导致。

8. `@DisableApiOperation` 的判断优先级高于 OpenAPI。
   - 即使某方法在源码中存在，只要被 `@DisableApiOperation` 禁用，前端不得显示为页面动作。
   - 如果 OpenAPI 暴露与 `@DisableApiOperation` 不一致，应记录不一致并进一步确认原因。

9. 前端页面开发前，应先整理有效控制器清单：
   - 控制器类名。
   - 是否 Spring Bean。
   - 是否抽象类。
   - 是否有 `@CRUD` 或继承 `@CRUD`。
   - 是否被条件注解启用。
   - 对应基础路径。
   - 对应菜单/路由路径。
   - OpenAPI 是否能辅助确认当前运行时暴露。

### 三、控制器优先级

1. 业务控制器优先级高于父类控制器。

2. 如果业务控制器没有重写某个父类方法：
   - 使用父类方法上的 Spring MVC 映射。
   - 使用父类方法上的 `@Operation`。
   - 使用父类方法上的 `@CRUD.ListTable`。
   - 使用父类方法上的 `@CRUD.Op`。
   - 使用父类方法参数和返回值作为页面字段来源。

3. 如果业务控制器重写了父类方法：
   - 业务控制器方法优先。
   - 父类方法作为缺省元数据来源。
   - 不能因为业务控制器方法未重复声明某些注解，就认为这些注解不存在。

### 四、方法级注解继承与覆盖

1. `@CRUD.ListTable` 和 `@CRUD.Op` 是方法级页面元数据，可以从父类控制器方法继承。

2. 判断某个控制器方法的前端页面元数据时，应同时检查：
   - 当前业务控制器方法。
   - 父类控制器同签名或同语义方法。
   - 被重写方法上的 Spring MVC 映射注解。
   - 被重写方法上的 `@Operation`。
   - 被重写方法上的 `@CRUD.ListTable`。
   - 被重写方法上的 `@CRUD.Op`。

3. 如果业务控制器没有重写某个父类方法：
   - 直接使用父类方法上的 Spring MVC 映射和 CRUD 注解。
   - 例如父类 `list()` 上有 `@CRUD.ListTable`，业务控制器未重写，则该列表页仍然有效。

4. 如果业务控制器重写了父类方法，但没有重新声明部分注解：
   - 应继承父类同方法上的缺失注解。
   - 例如业务控制器重写 `update()`，只声明了 `@CRUD.Op(visibleOn = "...")`，但没有重新声明 `@PutMapping` 或 `@Operation`，前端仍应继承父类 `update()` 的请求路径、HTTP 方法和操作名称。
   - 业务控制器只覆盖它显式声明的注解属性，其余未声明属性继续从父类方法继承。

5. 如果业务控制器重写方法并重新声明了同类注解：
   - 当前业务控制器方法优先。
   - 父类方法只作为缺省值来源。
   - 子类显式声明的 `@CRUD.Op` 属性覆盖父类 `@CRUD.Op` 对应属性。
   - 子类显式声明的 `@CRUD.ListTable` 属性覆盖父类 `@CRUD.ListTable` 对应属性。
   - 子类显式声明的 `@Operation`、`@RequestMapping` / `@GetMapping` / `@PostMapping` 等也覆盖父类对应信息。

6. 继承和覆盖应按“属性级合并”，不要简单地用子类注解整体替换父类注解。
   - 父类：`@CRUD.Op(confirmText = "None")`
   - 子类：`@CRUD.Op(visibleOn = "status == 'Draft'")`
   - 合并后应同时拥有：
     - `confirmText = "None"`
     - `visibleOn = "status == 'Draft'"`
   - 除非子类显式设置了同名属性，否则父类属性应保留。

7. 方法匹配应优先使用 Java 重写关系或方法签名。
   - 同名方法只是辅助判断。
   - 如果存在重载方法，应按参数类型、HTTP 映射和语义进一步区分。
   - 不应只按方法名粗暴合并所有注解。

8. `@DisableApiOperation` 仍然具有最终禁用优先级。
   - 即使方法从父类继承了 `@CRUD.Op` 或 `@CRUD.ListTable`，只要当前有效控制器被 `@DisableApiOperation` 禁用该方法或路径，前端不得展示对应页面或按钮。

9. 如果继承关系无法通过源码静态判断完整确认，可以结合 OpenAPI 辅助判断方法是否实际暴露；但页面元数据仍以控制器源码和注解继承/覆盖规则为主。

### 五、控制器方法参数决定页面字段

1. 页面字段必须通过控制器方法签名和源码分析确定。

2. 查询表单字段来自列表查询方法的参数，而不是固定命名的 `QueryXxxReq`。

3. 新增表单字段来自新增方法的参数，而不是固定命名的 `CreateXxxReq`。

4. 编辑表单字段来自编辑方法的参数，而不是固定命名的 `UpdateXxxReq` 或 `SimpleUpdateXxxReq`。

5. 删除、批量、业务操作表单字段来自对应操作方法的参数。

6. 如果方法参数是 DTO / BO / VO，则读取该类型字段及其注解。

7. 如果方法参数是基本类型、枚举、集合、`@RequestParam`、`@PathVariable` 等，则按参数本身和参数注解生成对应输入项。

8. 如果方法有多个参数，应按 Spring MVC 参数绑定语义区分：
   - 请求体参数。
   - 查询参数。
   - 路径参数。
   - 表单参数。
   - 分页参数。
   - 当前请求上下文参数。
   - 注入上下文参数。

9. 上下文注入类参数、分页参数、请求上下文参数等不应错误展示为业务表单字段。

10. 列表字段来自列表方法返回值中的数据项类型，或控制器/注解明确指定的列表实体/结果类型。

### 六、页面与按钮追溯要求

1. 每个菜单页面必须能明确追溯到：
   - 对应哪个控制器。
   - 对应哪个列表方法。
   - 对应哪些按钮方法。
   - 对应哪些禁用规则。
   - 对应哪些参数类型。
   - 对应哪些返回类型。

2. 每个页面至少要明确：
   - 列表来自哪个 `@CRUD.ListTable` 方法。
   - 页面级按钮来自哪些 `recordRefType=None` 或 `opRefTargetType=None` 方法。
   - 单条记录按钮来自哪些单条 `@CRUD.Op` 方法。
   - 批量按钮来自哪些批量 `@CRUD.Op` 方法。
   - 被禁用的方法为什么不展示。

### 七、字段规则应用顺序

1. 先根据控制器方法参数、返回值和源码注解确定字段来源。

2. 再按以下规则执行字段过滤、排序、组件、选项、上传、校验：
   - 《前端页面开发规则》
   - 《前端页面上传规则》
   - 《前端页面校验规则》

3. 如果控制器源码和 OpenAPI 展示不一致，以控制器源码为准。

4. 如果源码信息不足，应优先补充后端注解或与用户确认，不应通过 OpenAPI 猜测前端行为来绕过源码缺失。

### 八、管理后台接口开发补充规则

1. 管理后台前端的 API 接口封装，也必须按 Spring MVC 控制器独立划分。

2. 不允许把多个控制器的接口随意混在同一个通用 API 文件里长期维护。

3. 每个业务控制器对应的前端 API，应有明确的独立模块或文件归属。
   - 例如用户控制器相关接口归属用户 API 模块。
   - 字典控制器相关接口归属字典 API 模块。
   - 菜单控制器相关接口归属菜单 API 模块。
   - 不应把所有管理后台接口都堆到一个“大而全”的通用 API 文件里。

4. API 模块划分应以控制器为主，而不是以页面临时需求为主。
   - 页面可以组合调用多个 API 模块。
   - 但 API 文件本身应保持控制器边界清晰。

5. 如果业务控制器和父类控制器共同组成一个页面模块，API 归属仍以业务控制器对应模块为主。
   - 父类控制器方法只是该 API 模块的一部分来源。
   - 不应因为调用了父类通用 CRUD 方法，就把这些接口移到所谓“通用 API 大杂烩”中。

6. 前端调用接口时，应优先使用与控制器边界一致的 API 模块，不直接在页面里散写 `requestClient` 请求。
   - 页面内临时直调只能作为过渡，不应成为长期实现方式。

7. 通用能力可以抽象，但不能破坏控制器边界。
   - 例如可以有统一的请求层、错误处理层、上传层、权限层。
   - 但具体业务接口函数仍应按控制器归属组织。

8. 如果某些接口确实属于跨控制器公共能力，例如登录态、枚举、字典、上传、OAuth、RBAC 公共信息，可单独放在公共 API 模块。
   - 这类公共 API 必须有清晰理由，不能借口“通用”把本应独立的业务控制器接口重新混合。

9. 页面开发时，要能明确回答：
   - 当前页面对应哪个控制器。
   - 当前页面调用了哪些控制器接口。
   - 这些接口分别归属哪个前端 API 模块。

### 九、CRUD.Action 规则

1. `@CRUD.Op.action` 表示按钮点击时的主动作类型。

2. `@CRUD.Op.successAction` 表示接口成功后的后续动作。

3. `@CRUD.Op.failAction` 表示接口失败后的后续动作。

4. 同一个 `CRUD.Action` 枚举值，在 `action`、`successAction`、`failAction` 中都可能出现，但语义应结合当前阶段解释。

5. 前端必须按后端定义解释 `CRUD.Action`，不能擅自改成别的交互方式。

#### 默认动作规则

6. `action = Auto` 时，默认按 `Ajax` 处理。

7. `successAction = Auto` 时：
   - 如果主动作是 `Ajax`，默认按 `ReloadDataList` 处理。
   - 但如果方法语义明显是查看详情或返回结果对象用于展示，应按后端明确配置的展示动作处理，不应继续按 `ReloadDataList` 兜底。

8. `failAction = Auto` 时，默认按统一错误提示处理，不额外做跳转或容器动作。

#### 展示型动作

9. `ShowForm`
   - 用于展示接口成功返回结果中的对象详情。
   - 前端应基于响应结果中的 `data` 类型生成只读详情表单。
   - 不能复用当前列表页的创建表单、编辑表单或详情页表单。
   - `ShowForm` 默认只读，不带保存提交。
   - `ShowForm` 也应遵守详情表单规则：列宽一致、最大宽度不超过页面内容区域的 70%、长内容独占一行。
   - 如果返回的 `data` 不是对象，而是字符串、数组、图片地址、链接等，应按返回值语义降级为合适的只读展示方式，而不是强行套对象表单。

10. `ShowQrCode`
   - 用于把接口成功返回的结果数据按“二维码内容”处理。
   - 如果返回结果是字符串，默认按二维码原始内容生成二维码并展示。
   - 如果返回结果是对象，优先提取明确的二维码字段，例如 `qrCode`、`mfaQrCode`、`url`、`value` 等。
   - `ShowQrCode` 的主体是二维码，不是详情表单。

11. `ShowImage`
   - 用于把接口成功返回的结果数据按“图片资源”处理。
   - 如果返回结果是字符串，默认按图片地址、图片 data URL、base64 图片内容处理。
   - 如果返回结果是对象，优先提取图片字段，例如 `image`、`imageUrl`、`url`、`src`、`coverUrl` 等。
   - `ShowImage` 的主体是图片预览，不是详情表单。

12. `ShowVideo`
   - 用于把接口成功返回的结果数据按“视频资源”处理。
   - 如果返回结果是字符串，默认按视频地址处理。
   - 如果返回结果是对象，优先提取视频字段，例如 `video`、`videoUrl`、`url`、`src` 等。
   - `ShowVideo` 的主体是视频播放器或视频预览，不是详情表单。

13. `ShowIFrame`
   - 用于把接口成功返回的结果数据按“可 iframe 加载的页面地址”处理。
   - 如果返回结果是字符串，默认按 iframe 的 `src` 地址处理。
   - 如果返回结果是对象，优先提取 iframe 地址字段，例如 `url`、`src`、`iframeUrl`、`link` 等。
   - `ShowIFrame` 的主体是 iframe 页面，不是详情表单。

14. `ShowSchema`
   - 用于按 schema 或结构化数据方式展示结果。
   - 适用于结构化只读展示，不应默认等同于编辑表单。

#### 请求型动作

15. `Ajax`
   - 表示调用后端接口。
   - 这是最常见的业务按钮动作。
   - 结果再交由 `successAction` / `failAction` 处理。

16. `Jsonp`
   - 只有控制器明确这样定义时才使用。
   - 前端不能自行把普通接口当 JSONP 处理。

17. `Js`
   - 只有后端明确给出脚本语义时才使用。
   - 前端不能自行脑补脚本行为。

#### 导航型动作

18. `Link`
   - 用新窗口或新标签打开目标链接。

19. `Url`
   - 当前窗口跳转到目标地址。

20. `Prev`
   - 返回上一步。

21. `Next`
   - 进入下一步。

22. `Close`
   - 关闭当前对话框、抽屉或当前容器页签。
   - 必须结合当前容器上下文处理。

23. `Cancel`
   - 取消当前动作或关闭当前交互。
   - 语义偏中止，不等同于普通关闭。

#### 通讯型动作

24. `Tel`
   - 按电话号码动作处理，例如 `tel:` 跳转。

25. `Email`
   - 按邮箱动作处理，例如 `mailto:` 跳转。

26. `Copy`
   - 把结果数据复制到剪贴板。
   - 如果配置了 `resultActionData`，优先复制提取后的结果数据。

#### 数据刷新型动作

27. `ReloadDataList`
   - 重新加载当前列表页数据。

28. `UpdateData`
   - 更新当前页面上下文数据，而不是整页刷新。
   - 如果页面暂未支持局部更新，应保留扩展点，不能直接偷换成别的动作。

29. `Toast`
   - 展示轻提示。
   - 提示内容优先取后端返回消息。
   - 如果后端没有消息，再取按钮或操作默认成功文案。

#### resultActionData 规则

30. `resultActionData` 表示后续动作消费的数据来源。

31. 如果配置了 `resultActionData`，则 `successAction` / `failAction` 应优先使用提取后的结果数据。

32. 如果没有配置 `resultActionData`，默认使用接口返回的 `data`。

33. 如果接口返回没有 `data`，则按动作语义降级处理：
   - `ReloadDataList`：允许直接刷新
   - `Toast`：允许只提示
   - `ShowForm`：不应打开空表单
   - `Copy`、`Link`、`Url`：没有有效数据时不应执行

前端相关需求也必须按 OpenSpec 规范管理。涉及页面新增、交互调整、按钮行为变化、权限规则变化、接口适配变化、视觉改版时，必须先更新 `openspec/` 中对应需求，再开始编码。

前端开发流程统一按 superpowers 管理。默认采用结构化的需求拆解、计划执行、验证收口流程；不要跳过需求整理直接改页面。

前端项目统一使用语言：简体中文。页面文案、交互提示、占位文案、错误提示、注释、规则文档、说明文字、测试描述、开发输出等，默认都必须使用简体中文；仅在第三方字段名、标准协议名、框架保留关键字等不可翻译场景下保留原文。

请优先阅读以下后端代码，再开始前端页面开发：
- `com.levin.commons.ui.annotation` 包
- `admin-api/src/main/java/com/levin/oak/base/controller/BizArticleController.java`
- 对应的父类控制器，例如 `admin-api/src/main/java/com/levin/oak/base/controller/base/article/ArticleController.java`

前端按钮和页面操作的统一开发规范如下。

一、页面和菜单来源

1. 控制器类或其父类上有 `@CRUD` 注解，表示该控制器对应一个菜单模块或业务页面模块。

2. 模块名称、业务名称优先取控制器上的 `@Tag`、`@CRUD.label`、`@CRUD.desc` 等元数据；不要在前端重新发明一套名称。

3. 前端开发时必须同时检查业务控制器和其生成父类控制器。
   业务控制器中的重写方法，优先级高于父类同名方法。
   如果业务控制器没有重写，前端应继承父类方法上的页面和按钮定义。

4. `@DisableApiOperation` 禁止的接口，不得作为前端按钮、菜单动作或批量动作暴露出来。
   这条规则同时适用于当前类和继承链上的方法。
   例如 `@DisableApiOperation({"批量*"})` 时，所有“批量新增 / 批量更新 / 批量删除”都不得展示。

二、列表页来源

5. 控制方法上有 `@CRUD.ListTable` 注解，表示该方法对应一个列表页面。
   一个控制器下可以存在多个列表页面，前端不得假设一个模块只能有一个列表页。

6. 列表页的唯一标识使用 `@CRUD.ListTable.name`。
   未显式填写时，按默认值 `default` 处理。

7. 列表页标题、说明优先取以下顺序：
   `@CRUD.ListTable.label`
   `@Operation.summary`
   方法语义名

8. `@CRUD.ListTable.visibleOn` 是列表页显示条件。
   前端应把它作为后端定义的显示规则处理，不得把业务状态判断硬编码到页面里替代它。

三、按钮操作来源

9. 控制方法上有 `@CRUD.Op` 注解，表示该方法对应一个前端按钮操作。
   前端按钮必须来源于后端方法，不允许前端脱离控制器元数据自行定义业务按钮。

10. 按钮的基础信息优先按以下顺序解析：
   按钮名称：`@CRUD.Op.label`，为空时取 `@Operation.summary`
   按钮说明：`@CRUD.Op.desc`，为空时取 `@Operation.description`
   按钮图标：`@CRUD.Op.icon`
   按钮等级：`@CRUD.Op.level`

11. `@CRUD.Op.refListTable` 表示该按钮属于哪个列表页。
   前端必须把按钮挂到对应列表页，不允许把一个列表页的按钮错误展示到另一个列表页。
   未显式填写时，按默认值 `default` 处理。

12. `@CRUD.Op.recordRefType` 决定按钮显示位置：
   `None`：页面级按钮，展示在列表页顶部工具栏。
   `Single`：单条记录按钮，展示在表格行操作区。
   `Multiple`：批量按钮，展示在列表工具栏，并要求用户先选择多条记录。

13. 例如 `ArticleController.create()` 上的 `@CRUD.Op(recordRefType = CRUD.RecordRefType.None)`，前端应将“新增”放在页面级工具栏。

14. 例如 `BizArticleController.update()`、`auditCommit()`、`publish()` 等未显式指定 `recordRefType`，默认都是 `Single`，前端应展示在单条记录操作区。

四、按钮显隐和状态规则

15. `@CRUD.Op.visibleOn` 是按钮显示条件，前端必须按后端表达式语义处理。
   不允许把诸如 `status == 'Draft'` 之类的逻辑散落在各个页面里手写。

16. `visibleOn` 是显示条件，不是权限条件。
   前端应同时处理：
   业务显示条件：`visibleOn`
   权限条件：当前用户是否拥有该接口对应权限

17. `BizArticleController` 是标准示例：
   `update`、`auditCommit` 只在 `Draft` 或 `AuditRejected` 显示。
   `auditReject`、`auditApproved` 只在 `AuditPending` 显示。
   `publish` 只在 `Approved` 或 `Offline` 显示。
   `offline` 只在 `Published` 显示。
   `archived` 只在 `Offline` 显示。
   前端开发时应通过统一规则解释这些条件，不允许分别在每个页面写死。

18. `@CRUD.Op.confirmText` 表示操作确认提示内容。
   有值时，前端必须展示确认框。
   特殊值 `None` 表示显式不弹确认框。
   为空时，按“后端未要求确认”处理，不要强行追加业务确认框。

19. `@CRUD.Op.confirmTitle` 表示确认弹窗标题。
   有值时必须优先使用。

五、按钮行为规则

20. `@CRUD.Op.action` 表示按钮动作类型，前端必须优先按后端定义执行。
   常见值包括：`Ajax`、`Link`、`Url`、`ShowForm`、`ShowIFrame`、`ShowSchema`、`ReloadDataList` 等。

21. `@CRUD.Op.successAction` 和 `@CRUD.Op.failAction` 表示接口执行成功或失败后的后续动作。
   前端不得在没有后端定义的情况下自行发明成功后跳转、刷新、打开弹窗等行为。

22. `@CRUD.Op.viewContainerType` 用于界面型动作的容器类型选择。
   常见值包括：`Dialog`、`Drawer`、`Inline`、`Auto`。

23. `@CRUD.Op.actionData` 表示调用动作时提交的数据来源。
   一般默认使用当前记录数据或当前列表上下文数据。

24. `@CRUD.Op.resultActionData` 表示成功后动作所消费的数据来源。
   如果后端返回了结果对象，前端应优先使用该字段定义的结果数据，而不是自己拼装。

六、权限规则

25. 每一个按钮操作原则上对应后端的一个控制器方法，也对应一个后端权限点。
   前端按钮权限必须依据后端权限列表判断，不允许只靠菜单显隐代替按钮权限。

26. 当前用户权限列表必须通过 `RbacController.authorizedPermissionList()` 获取。
   只在以下时机重新加载：
   登录成功后
   浏览器整页刷新后
   普通页面路由切换时不要重复加载

27. 前端权限匹配必须使用后端提供的 `RbacPermissionMatchUtils.js/ts` 同等规则。
   不允许使用简单字符串相等判断替代。

28. 按钮是否展示，必须同时满足：
   接口未被 `@DisableApiOperation` 禁用
   当前列表页匹配 `refListTable`
   当前上下文满足 `recordRefType`
   `visibleOn` 条件为真
   当前用户拥有该接口对应权限

七、接口和前端实现约束

29. 前端按钮行为必须以控制器方法为准，优先读取以下信息：
   请求路径
   HTTP 方法
   `@Operation.summary`
   `@Operation.description`
   `@CRUD.Op`
   `@CRUD.ListTable`
   `@DisableApiOperation`

30. 前端不能因为当前页面暂时没有实现某种动作容器，就私自改成别的交互。
   例如后端要求 `Link` 时，不能擅自改成 Ajax。
   如果当前前端尚不支持某动作，应先在规则和代码中保留扩展点，再补实现。

31. 前端对于业务控制器重写的方法，要按重写后的方法重新解析。
   例如 `BizArticleController.create()` 虽然沿用父类“新增”入口，但业务控制器已经修改了新增时的默认状态，前端不应绕过该业务入口。

32. 前端页面开发时，若同一模块存在业务控制器和默认控制器，优先对接业务控制器暴露的接口，不直接绕过业务控制器去调用默认控制器接口。

八、开发落地要求

33. 以后新增或调整系统页面、工具栏按钮、表格行按钮、批量按钮时，必须先检查对应控制器的注解定义，禁止先写死前端再去反推后端。

34. 前端应尽量建设统一解析层，至少统一处理以下能力：
   列表页分组
   按钮分组
   `recordRefType`
   `refListTable`
   `visibleOn`
   `confirmText`
   `action/successAction/failAction`
   权限匹配

35. 不允许在每个页面重复写一套按钮显示规则；应优先修改统一解析层或共享组件。

36. 表单布局必须根据字段数量自动选择合适列数。
   非特殊情况下，如果表单字段展示后超过 7 行，应优先使用多列布局减少纵向滚动。
   多列布局最多不超过 4 列，避免表单过密、字段标签难以阅读或移动端难以适配。
   如果因为字段内容复杂、依赖关系强、需要大段文本编辑等原因必须保持单列，应在实现处保留清晰理由。

37. 如果后端控制器注解无法完整表达某个前端动作，先补充后端元数据，再实现前端；不要让前端长期依赖硬编码例外。

九、页面目录归组规则

38. 前端页面目录应按当前项目的模块包名归组存放，以便后续支持多个模块并存。

39. 模块目录名不直接按 Java 包路径展开，而应使用扁平化后的模块包名。
   包名中的 `.` 统一替换为 `_`。
   例如：`com.levin.oak.base` 应对应为 `com_levin_oak_base`。

40. 页面目录结构不应过深。
   不使用 `com/levin/oak/base/...` 这种多层目录展开方式。
   应在页面根目录下使用一层模块目录承载该模块下的页面。

41. 页面目录组织顺序应为：
   先按模块包名目录归组。
   再在模块目录下按具体页面资源建立独立页面目录。

42. 每个页面仍应保持自己的独立目录和页面实现文件。
   页面入口文件、`config.ts`、页面子组件、页面专属弹窗、页面专属工具函数等，都应归属在对应模块目录下。

43. 公共组件可以继续放在共享目录，但业务页面及其专属实现必须保留模块归属信息。

44. 这种模块包名归组方式应同时避免：
   不同模块同名页面冲突。
   路由命名冲突。
   页面实现长期平铺在同一层目录下导致维护混乱。

45. 后续新增模块时，应优先在新的模块包名目录下新增页面，而不是继续平铺到现有页面目录中。
