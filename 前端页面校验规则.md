# 前端页面校验规则

本文件用于单独维护管理后台前端页面中的表单校验规则。规则审核通过后，可作为《前端页面开发规则》的配套规范引用。

## 字段级校验规则

- 字段级校验以 DTO 字段上的 Java 验证器注解为准。
- 前端单字段校验规则应优先从后端 DTO 字段注解推导。
- 常见来源包括但不限于：
  - `@NotNull`
  - `@NotBlank`
  - `@NotEmpty`
  - `@Size`
  - `@Min`
  - `@Max`
  - `@DecimalMin`
  - `@DecimalMax`
  - `@Pattern`
  - `@Email`
  - `@Positive`
  - `@PositiveOrZero`
  - `@Negative`
  - `@NegativeOrZero`
  - 其他 Jakarta Bean Validation / Hibernate Validator 注解
- 如果 DTO 字段没有 Java 验证器注解，前端不应自行臆造必填、长度、格式等强约束。
- `@Schema(requiredMode = REQUIRED)`、`required = true` 可作为辅助信息，但字段级校验仍应优先以 Java 验证器注解为准。
- 前端校验信息应尽量使用后端注解语义生成，避免和后端实际校验不一致。

## 字段类型与业务校验的边界

- 字段类型只决定基础输入体验，不等价于业务校验。
- 例如 `String email` 字段如果没有 `@Email`，前端不应强行加邮箱格式校验，除非项目另有明确全局规则。
- 例如字段名包含 `phone/mobile`，如果 DTO 没有对应格式注解，前端可使用合适输入组件，但不应强行阻止提交。
- 类型转换校验除外：例如 Java 字段是 `Integer`，前端提交前必须保证值可转换为数字，否则应明确报错。

## 跨字段联合校验规则

- 跨字段联合校验来自表单、布局、分组或业务操作规则。
- 跨字段校验不应混入普通字段级规则中。
- 常见跨字段规则包括：
  - `newPassword` 与 `confirmPassword` 必须一致。
  - 起始时间不能晚于结束时间。
  - 某字段为某个枚举值时，另一字段必填。
  - 多个字段至少填写一个。
  - 同一分组内 `code` 不能重复。
- 这类规则应由表单配置、布局分组、操作定义或明确业务规则声明。
- 如果后端 DTO 没有表达跨字段约束，前端可根据表单/分组规则补充，但必须能说明规则来源。

## 集合和复杂对象字段校验

- 集合字段本身的必填、长度、元素数量应遵循 DTO 字段上的 Java 验证器注解。
- 集合元素如果是复杂对象，单条元素的字段级校验应来自元素 DTO / Java 类型的验证器注解。
- 集合内部的跨项校验，例如 `Dict.Item.code` 唯一，应来自后端业务校验、表单分组规则或明确业务规则。
- 前端可以提前做这类校验以提升体验，但不能和后端规则冲突。

## 前后端一致性原则

- 前端校验是用户体验优化，不应替代后端校验。
- 前端校验必须尽量与后端 DTO 注解和业务校验保持一致。
- 当后端校验规则变化时，前端应通过重新读取 DTO / OpenAPI / JsonSchema / 控制器源码同步更新。
- 如果前端无法确定某条校验是否存在，应宁可不加阻断式校验，交由后端返回错误。
