# 前端页面开发规则

1. 查询表单字段 label 和组件要并排显示，不要上下排列。

2. 页面实现过程中，应优先参考 `playground/src/views`,`docs/src/`  中 components, demos 和 examples 的页面结构、组件组合、间距、交互与代码组织方式。

   图片和文件上传相关规则独立维护在 `前端页面上传规则.md`，涉及上传字段识别、上传接口、回显预览、裁剪、截图粘贴等规则时必须同时遵守该文件。

   表单校验相关规则独立维护在 `前端页面校验规则.md`，涉及字段级校验、跨字段联合校验、集合和复杂对象字段校验时必须同时遵守该文件。

3. 页面标题和表单标题应分层处理，不允许直接复用页面标题生成创建、编辑、详情表单标题。

   标题规则：

   - 菜单标题、页面标题、标签页标题、面包屑标题可以保留“管理”后缀，例如：`用户管理`、`字典管理`。
   - 创建表单标题应使用：`新增{业务对象名}`，不带“管理”后缀，例如：`新增用户`、`新增字典`。
   - 编辑表单标题应使用：`编辑{业务对象名}`，不带“管理”后缀，例如：`编辑用户`、`编辑字典`。
   - 详情表单标题应使用：`{业务对象名}详情`，不带“管理”后缀，例如：`用户详情`、`字典详情`。
   - `successAction = ShowForm` 这类结果展示，不应套用页面标题或通用详情标题，应优先使用操作语义标题。
   - `ShowForm` 结果展示标题优先顺序为：
     - `@CRUD.Op.label`
     - `@Operation.summary`
     - 返回对象类型上的 `@Schema.title`
   - 例如 `MFA二维码`、`查看我的MFA二维码` 这类操作，应直接使用操作标题，而不是 `用户详情` 或 `编辑用户`。

4. 所有表单字段必须按统一的业务优先级排序，不允许只按 OpenAPI schema 原始字段顺序展示。

   字段排序优先级：

   | 顺序 | 字段规则 | 备注 |
   | --- | --- | --- |
   | 1 | `tenantId` | 查询表单整体排序基数会额外加 40；列表/表格中固定左侧，建议宽度 150px。 |
   | 2 | `orgId` | 查询表单整体排序基数会额外加 40；列表/表格中固定左侧，建议宽度 150px。 |
   | 3 | 主键字段（`@Id`） | 列表/表格中固定左侧，建议宽度 150px，支持复制；新增/编辑中通常隐藏或按提交规则处理。 |
   | 4 | 名称/编码模糊查询字段 | 查询表单中 `containsName`、`containsCode` 等名称和编码的模糊查找字段，排序优先级仅低于租户和 ID 类字段。 |
   | 5 | 名称类字段 | 包括 `name`、以及字段名等于或后缀为 `title`、`label`。 |
   | 6 | 手机号/邮箱字段 | 手机号优先于邮箱展示；字段位置在编码字段前。 |
   | 7 | 编码类字段 | 包括 `code`、以及字段名以 `Code` 结尾。 |
   | 8 | 类型类字段 | 包括 `type`、以及字段名以 `Type` 结尾。 |
   | 9 | 状态类字段 | 包括 `status`、`state`、以及字段名以 `Status`、`State` 结尾。 |
   | 10 | 时间/日期字段 | `Pay/Refund` 优先，其次 `Create/Add/New`，再其次 `Expired`，再其次 `Update/Modify`，其他时间字段靠后；小于/小于等于类范围条件排在同组后面。 |
   | 11 | 其他 `*Id` 字段 | 普通关联 ID 字段。 |
   | 12 | 普通字段 | 默认位置。 |
   | 13 | 大文本/编辑器字段 | `textarea`、`editor` 等复杂输入靠后。 |
   | 14 | 描述/备注字段 | 包括 `desc`、`remark`、以及字段名以 `Desc`、`Remark` 结尾，最后展示。 |

5. 查询表单字段在排序前需要执行查询场景过滤，避免把不适合查询的字段展示出来。

   查询表单应排除或转换以下字段：

   - 排除通用控制字段：`orderBy`、`orderDir`、`selectColumns`、`pinyinName`、`containsPublicData`、`containsOrgPublicData`、`tenantShared`、`orgShared`、`optimisticLock`。
   - 排除操作日志字段（如 `actionLog`），不展示、不参与查询提交。
   - 排除密码、密钥、图片、文件、CSS、JSON、大文本、LOB、复杂对象、关联 `Fetch` 字段。
   - 排除组织 ID 字段 `orgId`，但保留租户、组织、用户等明确可选的选择字段规则。
   - 排除创建时间/更新时间等继承基础字段，除非已经提供明确的范围查询字段。
   - 如果存在 `containsXxx`，则原始 `xxx` 字段不重复展示。
   - 如果存在 `inXxx` 或 `notInXxx`，则原始 `xxx` 字段不重复展示；`in` 优先级高于 `notIn`。
   - 如果存在 `betweenXxx`，则同一字段的 `lt/gt/lte/gte/in` 类查询不重复展示。
   - 查询表单中，成对出现的“开始时间 / 结束时间”字段应合并成一个时间范围选择器，不分成两个独立输入框。
   - 查询表单中的 `textarea` 统一降级为单行输入，避免查询区高度失控。
   - 查询表单中的 `input-array` 统一降级为 `input-text`，数组值由逗号或换行输入。

6. 新增/编辑表单字段也必须执行场景过滤。

   - 非展示表单排除只读字段（`@Schema(accessMode = READ_ONLY)`）。
   - 排除操作日志字段（如 `actionLog`），不展示，也不提交。
   - 排除创建时间、修改时间、更新时间字段（如 `createTime`、`lastUpdateTime`、`updateTime`），不展示，也不提交。
   - 新增表单中 `enable`、`editable` 默认值为 `true`。
   - 新增/编辑表单忽略 `orgIdList` 和树节点路径 `nodePath` 等不适合直接编辑的系统字段。
   - 乐观锁 `optimisticLock` 在编辑/提交场景应隐藏但保留提交能力。
   - 单行关联操作如果需要 ID，应把 ID 作为隐藏字段加回表单。
   - 文件路径、图标、图片等字符串类型字段，在新增/编辑表单中应使用上传组件，不使用普通文本输入。
   - 文件路径、图标、图片等字符串集合类型字段，在新增/编辑表单中应使用支持多文件的上传组件。
   - 图片类字段默认应支持裁剪；除非字段或页面明确声明不裁剪。
   - 详情、备注、描述等长文本字段，如果文本长度大于 256，应使用多行文本组件，并在多列表单中独占一整行。
   - 字段类型是集合时，应在多列表单中独占一整行，并支持动态添加、修改和删除集合元素。
   - 字段类型是 `Map` 或其子类，且字段语义不是图片或文件时，默认使用 Json 编辑器，并在表单中独占一整行。
   - 字段类型是单个复杂 DTO 对象，且不是集合类型时，应独占一整行，并使用独立容器包裹后展开其 DTO 字段，不应只渲染成一个普通输入框。
   - 基础类型集合字段如果元素类型是 `String`，且字段语义是标签、关键字、模块、角色编码、域名等基础字符串列表，应使用 tags 输入组件，支持动态新增、编辑、删除单个字符串项。
   - 基础字符串集合字段如果语义是图片或文件资源列表，应按 `前端页面上传规则.md` 使用多文件或多图片上传组件，不使用普通 tags 输入。
   - 集合字段如果是图片类型，应使用图片上传/图片管理组件。
   - 集合字段如果是文件类型，应使用文件上传/文件管理组件。
   - 图片和文件上传必须通过 `FileStorageController`，不要自行定义上传接口。
   - 集合字段如果元素是复杂 DTO 对象类型，应独占一整行，并默认使用“列表 + 元素编辑器”的方式编辑，而不是直接展开成大块嵌套表单。
   - 复杂 DTO 集合的列表区应展示每个元素的关键字段摘要，例如名称、编码、状态、备注等适合识别该元素的字段。
   - 复杂 DTO 集合必须支持新增元素、编辑元素、删除元素。
   - 编辑单个复杂 DTO 元素时，应通过独立编辑容器完成，例如弹窗、抽屉或子表单容器；编辑完成后回写到当前集合字段，提交时保持数组对象结构不变。
   - 复杂 DTO 集合不应退化为纯 JSON 文本输入，除非页面明确指定使用 Json 编辑器。
   - 复杂 DTO 集合中的关键字段摘要和元素编辑方式，必须由页面实现显式指定，不允许运行时猜测。
   - 字段上存在 `@JsonSchemaEditor` 注解时，优先按该注解声明的逻辑处理组件、schema 和编辑行为，上述默认规则不覆盖 `@JsonSchemaEditor`。
   - `@JsonSchemaEditor.title` 为空时，标题默认取字段 `@Schema.title`。
   - `@JsonSchemaEditor.desc` 为空时，描述默认取字段 `@Schema.description`。
   - `@JsonSchemaEditor.jsonSchema` 支持三种格式：
     - Java 类名，如 `com.test.UserAddress`，前端必须通过 `JsonSchemaController` 获取该类型的 JSON Schema。
     - URL，如 `/sys/jsonSchema/userAddr.json`，前端应按远程 schema 地址加载。
     - 同类属性引用，如 `:attrName`，表示从当前 DTO 同一个对象的属性中读取 schema。
   - 前端获取 Java 类 JSON Schema 必须通过当前模块 API 层调用 `JsonSchemaController.genJsonSchema`。
   - 请求体使用 `{ "typeGenericStr": "Java类名或泛型类型字符串" }`。
   - 接口返回数据中的 `jsonSchema` 字段为表单 schema 内容；结果可按 `typeGenericStr` 缓存。
   - `@JsonSchemaEditor.inlineMode=true` 时，动态表单内联铺开展示。
   - `@JsonSchemaEditor.inlineMode=false` 时，动态表单应通过弹窗、抽屉或子表单容器展示，避免主表单过长。
   - `@JsonSchemaEditor.columns=-1` 表示自动列数；自动列数按“字段数约每 7 行一列”计算，但不能超过父表单布局允许的最大列数，也不能超过项目规则的 4 列上限。
   - `@JsonSchemaEditor.columns>0` 时，优先使用注解指定列数，但仍不能超过父表单布局和项目 4 列上限。
   - JsonSchema 动态表单编辑复杂字段或复杂集合元素时，每个元素应支持新增、修改、删除；提交时保持对象或数组原始结构，不应转成逗号字符串。
   - 在详情、创建、编辑这三类表单中，同一表单内各列的列宽应保持一致，不允许出现某一列明显更宽、另一列明显更窄的随意布局。
   - 当表单使用多列布局时，应采用等宽列布局。2 列时两列等宽，3 列时三列等宽，4 列时四列等宽；只有字段被规则明确要求独占一行时，才允许跨列。
   - 详情表单、创建表单、编辑表单在同一页面模块中，应尽量保持一致的列宽策略和整体宽度策略，避免三者视觉结构差异过大。
   - 在详情、创建、编辑表单中，表单列宽不应超过 `480px`。
   - 当表单使用多列布局时，每列应保持等宽，并且单列宽度上限为 `480px`。
   - 表单容器最大宽度不应超过页面内容区域宽度的 70%；超过时应限制最大宽度，而不是继续横向拉伸。
   - 页面宽度增大时，应优先保持表单主容器最大宽度限制、列宽一致和单列宽度不超过 `480px`，而不是继续横向拉大输入框。
   - 当页面可用宽度较小时，应优先减少列数，而不是突破 70% 宽度限制；优先从 4 列降到 3 列，再从 3 列降到 2 列，必要时降到 1 列。
   - “表单最大宽度不超过页面宽度的 70%”适用于详情、创建、编辑表单主容器，不适用于查询表单、表格、图片预览弹窗、代码编辑器等特殊区域。
   - 对于独占一行的字段，例如大文本、JSON、上传区、子表单、复杂对象集合，可以占满表单可用宽度，但仍应受表单主容器最大宽度限制，不适用单列 `480px` 上限。
   - 如果某个页面因业务需要必须突破 70% 或采用非等宽列布局，应有明确例外理由，且应尽量局部化，不作为默认实现。

7. 组件类型选择必须按字段语义映射为合适的前端组件。

   | 字段/类型规则 | 组件语义 | 本地 Vue 推荐组件 |
   | --- | --- | --- |
   | 默认字符串 | 单行文本 | `Input` |
   | 字符串长度大于 255 或 LOB | 多行文本 | `Input.TextArea` |
   | 数字 | 数字输入 | `InputNumber` |
   | 布尔 | 开关 | `Switch` |
   | 枚举 | 下拉选择 | `Select` |
   | 日期时间 | 日期时间选择 | `DatePicker` 或日期时间组件 |
   | 日期 | 日期选择 | `DatePicker` |
   | 时间 | 时间选择 | `TimePicker` |
   | 范围日期时间 | 日期时间范围选择 | `RangePicker` |
   | 省/市/区 | 城市/地区选择 | 级联选择或城市选择组件 |
   | 租户 ID | 远程租户选择 | 远程 `Select` |
   | 组织 ID / 组织 ID 列表 | 授权组织树选择 | `TreeSelect` |
   | ownerId | 远程用户选择 | 远程 `Select` |
   | JSON | JSON 编辑 | JSON 编辑器或 `Input.TextArea`，必须校验 JSON |
   | 图片 | 图片上传 | 图片上传 |
   | 文件 | 文件上传 | 文件上传 |
   | HTML | 富文本编辑 | 富文本编辑器 |
   | CSS | CSS 代码编辑 | 代码编辑器 |
   | 密码 | 密码输入 | `Input.Password` |
   | 颜色 | 颜色选择 | 颜色选择器 |
   | URL | 单行文本 + URL 校验 | `Input` + URL 校验 |
   | 手机号 | 单行文本 + 手机号校验 | `Input` + 手机号校验 |
   | 邮箱 | 单行文本 + 邮箱校验 | `Input` + 邮箱校验 |
   | 身份证 | 单行文本 + 身份证校验 | `Input` + 身份证校验 |
   | 基础类型集合 | 数组输入 | 多行文本或 tags/select，提交时转数组 |
   | 子表单/对象集合 | 子表单或分组表单 | 子表单、分组表单或抽屉/弹窗编辑 |
   | 带 `@JsonSchemaEditor` 注解的字段 | JsonSchema 动态表单 | 按注解配置渲染 JsonSchema 动态表单 |

   文件上传规则：

   - 前端文件、图片上传必须通过 `FileStorageController`。
   - 上传接口路径由当前模块 API 层提供，不应在页面规则中写死某个模块的 API 前缀。
   - 单文件上传接口优先使用 `FileStorageController.uploadSingleFile`。
   - AMIS 兼容上传使用 `FileStorageController.uploadSingleFile` 并携带 `amisUpload=true`。
   - 多文件上传使用 `FileStorageController.uploadFiles`。
   - 获取 OSS 上传凭证使用 `FileStorageController.getOssUploadCredential`。

8. 第三方 OAuth 授权必须通过 `OAuthController`，不要在前端自行拼第三方平台授权地址。

   - 获取支持的第三方平台列表：`GET /oauth/getSupportedPlatforms`。
   - 发起第三方平台授权：`GET /oauth/authorize/{platform}`。
   - 授权回调地址：`GET /oauth/callback/{platform}`。
   - 发起授权时可传 `callbackUrl`，用于授权完成后回到前端指定页面；如果不传，后端会优先使用当前请求来源页面。
   - 授权成功后，后端会写入 `Authorization`、`AccessToken`、`Token` 响应头，并重定向回回调页面。
   - 前端页面只负责跳转到后端 OAuth 授权入口和处理回跳后的登录态，不直接调用第三方平台 SDK 地址。

9. 列表展示字段也必须按排序规则展示，并按字段语义选择展示方式。

   - 数字、枚举、日期时间、名称、手机号、邮箱字段可排序。
   - JSON、HTML、子表单等复杂字段默认不在列表直接展示。
   - 图片字段用图片预览组件。
   - 文件字段用下载动作。
   - URL 字段用链接组件。
   - 普通长文本在列表中应截断并提供 tooltip，不直接撑开表格。

10. 带 `@Options` 注解或具备选择语义的字段必须按选择型字段处理。

   选择型字段基础规则：

   - 组件使用下拉选择或树选择，不使用普通文本输入。
   - 查询表单中，选择型字段默认应加入查询条件。
   - 选择型字段默认使用单选组件。
   - 当 Java 字段类型是集合对象时，选择组件必须支持多选。
   - 当查询字段带 `In` / `NotIn` 查询语义时，查询表单中必须按多选处理。
   - 对于枚举类型字段，如果查询 DTO 中存在带 `@In` 的 `inXxx` 字段，应优先展示 `inXxx` 多选查询字段，而不是普通 `xxx` 单选字段。
   - `columnMapExpr` 用于覆盖选项值和显示字段：第 1 项为 `valueField`，第 2 项为 `labelField`，第 3 项为描述字段。
   - 如果没有显式 `columnMapExpr`，默认按选项来源推导 `valueField` 和 `labelField`。
   - `reloadOn` 用于声明重新加载选项的触发条件，通常只对远程 API、实体对象或 Dao 查询对象选项有效。

   选项来源优先级：

   | 优先级 | 来源规则 | 前端处理 |
   | --- | --- | --- |
   | 1 | `items` 固定选项 | 直接使用本地静态 options。 |
   | 2 | `dictCode` | 从系统字典接口加载；默认 `valueField=code`，`labelField=name`。 |
   | 3 | `refTargetType` 为枚举类 | 通过 `EnumController` 获取枚举选项。 |
   | 4 | `refTargetType` 为实体类或 Dao 查询对象 | 从目标对象列表接口加载；如果实体支持启用状态，默认附加 `enable=true`；支持搜索。 |
   | 5 | `loadApi` | 作为自定义远程选项接口；如果内容是 JSON，则按完整 API 配置处理；否则按 URL 字符串处理。 |

   如果同一个字段同时配置了多个来源，按上表优先级选择第一个可用来源。

   字典选项规则：

   - `dictCode` 对应系统字典编码。
   - 前端获取数据字典必须通过 `BizDictController.retrieveByCode` 方法。
   - 数据字典接口路径由字典控制器 API 模块提供，不应在页面规则中写死某个模块的 API 前缀。
   - 如果 `dictCode` 是 `$@`，`$@` 不是具体字典编码，只是动态推导占位符，必须根据当前 DTO 对象上下文推导真实字典编码。
   - `dictCode="$@"` 的推导规则：
     - 如果字段所属 DTO 上存在目标实体信息，则字典编码为：`目标实体完整类名.字段名`。
     - 如果当前字段是 `inXxx` 或 `notInXxx` 这类查询字段，应先去掉 `in` / `notIn` 前缀，再把剩余字段名首字母转小写后参与编码。
     - 如果无法获得目标实体信息，则字典编码为：`DTO名称.字段名`。
   - 字典接口返回项默认从返回数据的 `itemList` 中取，作为前端 options。
   - 字典选项默认 `valueField=code`，`labelField=name`。
   - 字典选项通常一次性加载完整 options，搜索应在前端 JS 中按 `label` 匹配；可用 `value` 兜底，但不能只按 `value` 匹配。

   枚举选项规则：

   - 前端获取枚举必须通过 `EnumController`。
   - 枚举选项应优先通过当前模块 API 层调用 `EnumController.enums` 一次性获取全部枚举，并缓存到应用内存中。
   - 应用级枚举缓存只在当前网页生命周期内有效，刷新整个网页后重新加载。
   - 如果全量枚举缓存中不存在目标枚举，再调用单个枚举接口，并把返回结果补写回同一个应用级缓存。
   - 单个枚举类接口由 `EnumController` 提供，具体路径由对应模块 API 层维护。
   - 返回数据中的 `options` 作为前端 options。
   - 枚举选项默认 `valueField=value`，`labelField=label`。
   - 枚举值使用枚举 `name()`，不要使用 `toString()`。
   - 枚举选项通常一次性加载完整 options，搜索应在前端 JS 中按 `label` 匹配；可用 `value` 兜底，但不能只按 `value` 匹配。

   实体关联选项规则：

   - 远程接口为目标实体的列表接口。
   - 搜索参数名应优先根据目标实体控制器 `list` 接口的查询 DTO 推导。
   - 推导搜索参数名时，模糊查询字段优先，例如存在 `containsName` 时优先使用 `containsName`。
   - 如果目标查询 DTO 不存在可用的模糊查询字段，再退回使用 `name`。
   - 如果注解显式提供 `searchParamName`，应使用该参数名作为覆盖值。
   - 用户在选择框输入搜索词时，应重新调用远程 options 接口，不应只在前端过滤当前已加载 options。
   - 远程搜索时最终请求参数应为：`defaultParams + { [searchParamName]: 用户输入的搜索词 }`。
   - `valueField` 默认使用目标实体主键字段。
   - `labelField` 按 `name`、`label`、`desc` 的顺序自动选择第一个存在字段。
   - 如果目标实体是树形结构，优先使用树选择语义；默认不展示默认图标。
   - 如果当前字段是集合类型，必须开启多选。
   - 如果 `defaultParams` 存在，应附加到远程选项接口请求中。
   - 如果目标实体支持启用状态，默认附加 `enable=true`，且搜索时不能丢失。

   自定义 API 选项规则：

   - `loadApi` 指向远程接口时，不应只在前端过滤当前已加载 options。
   - 用户在选择框输入搜索词时，应重新调用 `loadApi`。
   - 搜索词必须按 `@Options.searchParamName` 作为 URL 参数传给后端；如果 `searchParamName` 为空或未配置，优先根据接口参数推导模糊查询参数名，再退回使用 `name`。
   - 远程搜索时仍必须携带 `@Options.defaultParams`。
   - 如果 `@Options.reloadOn` 存在，应按其表达式监听依赖字段变化并重新加载选项。

   固定选项规则：

   - `items` 中每一项按 `value||label||desc` 解析。
   - `value` 必须存在。
   - `label` 为空时可用 `value` 兜底展示。
   - `desc` 可用于 tooltip、说明或扩展展示。
   - 固定选项搜索应在前端 JS 中按 `label` 匹配；可用 `value` 兜底，但不能只按 `value` 匹配。

   搜索参数编码规则：

   - Java 类名、字典编码、搜索词等作为 URL 参数传递时必须进行 URL 编码。
   - Java 内部类名可能包含 `$`，例如 `com.xxx.User$Sex`，必须编码为 `%24` 后传递。
   - 前端不应手工拼接未编码的 Java 类名到 URL 中。
