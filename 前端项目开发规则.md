# 前端项目开发规则

涉及前端页面、浏览器页面、交互流程、视觉状态、下载行为、路由可访问性或用户当前页面问题的验证, 建议开启playwright 测试。

## 前端界面验证和本地烟测规则

1. 涉及登录页、布局、主题、租户站点 UI 配置、菜单、路由、权限、按钮显隐、表格渲染或页面交互的改动，不能只用 `build`、`typecheck`、单元测试证明完成；必须补充真实页面验证，至少覆盖受影响页面能打开、关键接口返回正常、关键 UI 区域可见、控制台没有新增错误。

2. 浏览器验证优先使用用户当前的 in-app browser。若 in-app browser 工具不可用、连接超时或被用户现场占用，可以使用项目本地 Playwright/Chromium 作为补充验证，但最终说明必须明确标注这是补充验证，不能声称已经代表用户当前浏览器现场。

3. 本地已有前端开发服务或预览服务在目标端口运行时，优先复用当前服务并记录实际访问地址；不要为了验证随意重启服务。只有确认当前服务不可用、版本明显过旧或端口不是目标应用时，才启动或重启前端服务。

4. 登录态页面烟测不得硬编码历史 accessToken。需要登录后访问页面时，应优先走真实登录链路：调用验证码接口获取当前验证码，再调用登录接口获取本次 accessToken；若测试使用本地存储注入登录态，存储 key 必须从当前应用 `VITE_APP_NAMESPACE`、应用版本和环境名推导，不能复制旧项目名或旧命名空间常量。

5. 登录页和租户站点 UI 配置相关验证必须检查 `tenantSiteInfo` 接口。只要页面调用了该接口，就必须确认接口返回成功，并确认返回的租户站点信息或 `uiExInfo` 中的界面配置已经参与当前界面处理；不能只确认接口被调用而不确认界面应用效果。

6. 服务端下发的后台基础界面配置使用 `uiExInfo.admin-ui-base-setting`。验证上传和回显时，应同时检查该 key 下的 `setting` 和 `preferServerSetting`；用户登录后如果调用了 `tenantSiteInfo`，前端必须按返回数据处理当前界面配置。

7. 页面烟测断言应优先使用用户可感知的稳定信号，例如当前 URL、页面标题、菜单文本、面包屑文本、表格列标题、按钮文本、空数据文案、接口响应状态和业务数据区域可见性。不得把单个组件库内部 class 作为唯一断言依据，例如只找 `.ant-table-wrapper` 来判断表格是否存在。

8. 通用 CRUD 表格可能由 Ant Table、VXE Table、HTML table 或封装组件实现。烟测表格可见性时，应使用页面业务语义和表格区域的稳定结构综合判断，例如列表标题、查询区、列标题、空数据区、`role="table"` 或已约定的测试标识；不要把某个历史 UI 实现当成唯一标准。

9. 页面标题、菜单标题和测试断言必须来自后端源码注解或当前页面真实文案。涉及控制器页面名称时，优先使用控制器 `@Tag.name`、`@CRUD.title`、`@Operation.summary` 等后端元数据或页面当前实际标题，不得通过“去掉管理”等字符串规则推导。

10. 旧烟测失败时，必须先查看失败截图、错误上下文、控制台错误和网络请求，再判断是页面回归还是测试断言过期。不能因为测试没找到旧 class、旧标题或旧命名空间，就直接判定页面功能失败。

11. 验证报告应区分三类结论：
    - 构建/单测结论，例如 `vitest`、`build` 是否通过。
    - 浏览器界面结论，例如页面是否打开、关键 UI 是否可见、控制台和网络是否正常。
    - 环境或测试脚本问题，例如浏览器依赖缺失、in-app browser 不可用、旧测试选择器或旧登录态 key。

12. 如果为了运行 UI 烟测补装浏览器依赖、生成测试 trace、截图或视频，应在报告中说明；这些产物只作为验证证据，不应混入业务代码提交。

## 前端模块打包和发布规则

- 所有前端可发布模块都必须遵循本规则，包括公共基础包、UI 包、effects 包、业务模块包和应用框架包。
- 发布包可以包含 `src` 源码，`src` 只用于调试、源码映射和问题排查；不得把 `src` 作为私服发布后的稳定运行入口或稳定类型入口。
- `package.json` 中面向发布消费者的 `main`、`module`、`types` 和 `publishConfig.exports` 必须指向 `dist` 产物；有子路径导出的模块，子路径的 `publishConfig.exports` 也必须指向对应 `dist` 文件。
- 开发态允许在普通 `exports` 中保留 `development -> src`，以便 monorepo 本地开发调试；但发布态必须通过 `publishConfig.exports` 覆盖为 `dist`，避免下游模块解析到源码入口。
- 包内 `files` 应同时包含 `dist` 和 `src`：`dist` 提供稳定运行和类型产物，`src` 提供调试源码。除非该包本身没有源码目录，否则不得只发布 `dist` 而丢失 `src`。
- 依赖下游包的模块不得只导出 `dist/index.mjs` 而缺少对应 `types`；每个对外导出的入口都应提供可解析的 `.d.ts` 声明。
- 构建配置必须保证 `dist` 中存在所有发布入口引用的文件，包括根入口、子路径入口、样式入口、Vue 组件入口和声明文件入口。
- 样式源码、Vue 单文件组件或其它调试资产如果不适合由 `unbuild`/`mkdist` 编译，应通过构建脚本原样复制到 `dist`，但发布入口仍应指向稳定的 `dist` 路径。
- 打包或发布前必须至少验证：
  - 相关包的 `build` 通过。
  - `publishConfig.exports` 不再指向 `src`。
  - `npm pack --dry-run` 能看到 `dist` 和 `src` 都进入包内。
  - `publishConfig.exports` 中引用的 `dist` 路径在包内真实存在；通配符导出也必须能匹配到实际文件。

## 前端多`@CRUD.ListTable`页面开发规则

- 当同一个控制器或页面存在多个`@CRUD.ListTable`时,必须按多列表页面处理.
- 单个`@CRUD.ListTable`不展示Tab;多个`@CRUD.ListTable`必须展示悬浮Tab.
- 悬浮Tab是真正浮层,不占用页面布局空间;初始位置为页面内容区`top:20px; left:20px`;背景和边框使用框架选中态语义色,例如`bg-primary/5`、`border-primary`、`text-primary`对应的主题变量,不得定义具体颜色;不透明;支持鼠标拖动改变位置;Tab名称纵向排列;支持展开和收缩,收缩后显示为圆形图标按钮;不属于表格内部工具栏,不随表格横向滚动. 本条仅适用于多`@CRUD.ListTable`页面的悬浮Tab,不属于公共可拖拽浮动面板基础设施,不得与 `DraggableFloatingPanelHost` / `useDraggableFloatingPanels` 混用.
- Tab文案优先使用`@CRUD.ListTable.title`,其次使用`@CRUD.ListTable.name`,再其次使用列表方法名或稳定业务兜底名.
- 每个Tab代表一个独立的ListTable实例,不是同一张表格的筛选视图. 默认情况下,不同Tab之间不得共享任何列表级配置、状态或操作.
- 每个Tab必须独立维护`listPath`、`tableName`、查询条件、分页状态、排序状态、选中行、列显示/隐藏配置、列固定配置、表格列定义、行操作、批量操作、工具栏操作和表格数据加载状态.
- 操作按钮必须按具体ListTable归属展示. `@CRUD.Op`生成的按钮统一按`opRefTargetListName`归属过滤;新增、详情、编辑、删除等内置CRUD业务操作也必须归属到具体ListTable,不得作为页面级全局按钮跨Tab展示.
- 未显式归属到当前ListTable的操作,在当前Tab中一律不展示;未配置归属的默认CRUD操作只属于默认ListTable.
- 只有在配置中显式声明为共享的字段、查询项或操作,才允许跨Tab复用;生成器不得隐式复用默认ListTable的配置到其它Tab.
- 切换Tab时必须加载当前Tab自己的列表接口和表格配置,不得复用上一个Tab的分页、选中行、列设置、查询条件或操作按钮.
- 开发者或AI辅助实现具体页面时,如果发现多个`@CRUD.ListTable`,必须在页面配置中明确维护`listTables`配置.

## 前端表单通用提交规则

- 表单字段使用选择类组件时,如果用户没有选择任何值,提交时不得提交该字段.
- 选择类组件包括下拉选择、角色选择、树选择、级联选择、标签选择等以候选项选择为主的控件.
- 空选择值包括`undefined`、`null`、空字符串和空数组.
- 普通输入类字段、开关字段、数字字段、文本字段不套用本规则,应按各自字段类型的序列化规则处理.

## 前端页面调用接口规则

- 非特殊情况下,页面组件的所有接口调用都必须通过当前模块的`api`目录下导出的接口、服务或方法完成. 模块化业务页面优先使用对应`src/modules/<module>/api`目录;仅跨模块通用能力才放入应用或公共包的`src/api`目录.
- 页面组件默认不得直接使用`requestClient`、通用`fetchCrudList`、或`views`目录内自建的接口封装访问任何后端接口.
- 如果页面缺少所需接口,优先在`api`目录补充对应服务方法,再由页面调用该服务方法.
- 确有特殊情况需要绕过`api`服务层时,必须在代码附近说明原因,并保持调用范围局部、可替换.

## 前端页面和组件主题颜色规则

- 所有前端页面开发、通用组件开发和业务组件开发都必须遵循本规则.
- 非特殊情况下,页面和组件不得自定义具体颜色值,包括`#fff`、`#1677ff`、`rgb(...)`、`rgba(...)`、固定`hsl(...)`等硬编码颜色.
- 当前框架颜色以`packages/@core/base/design/src/design-tokens`中的CSS变量为主,并由偏好设置在运行时动态更新. 页面和组件的自定义样式优先使用这些变量或它们对应的Tailwind语义类.
- 可直接使用的基础语义变量包括`--background`、`--background-deep`、`--foreground`、`--card`、`--card-foreground`、`--popover`、`--popover-foreground`、`--muted`、`--muted-foreground`、`--accent`、`--accent-foreground`、`--accent-hover`、`--secondary`、`--secondary-foreground`、`--border`、`--input`、`--input-background`、`--ring`、`--header`、`--sidebar`、`--sidebar-deep`、`--overlay`、`--overlay-content`、`--heavy`、`--heavy-foreground`等.
- 可直接使用的状态/业务语义变量包括`--primary`、`--primary-foreground`、`--success`、`--success-foreground`、`--warning`、`--warning-foreground`、`--destructive`、`--destructive-foreground`、`--info`、`--info-foreground`等.
- `primary`、`success/green`、`warning/yellow`、`destructive/red`会按用户主题动态生成`50`到`700`色阶,并提供`hover`、`active`、`text`、`text-hover`、`text-active`、`border`、`border-light`、`background-light`、`background-lighter`、`background-lightest`等Tailwind语义类. 例如`text-primary`、`text-primary-hover`、`bg-primary-background-lightest`、`border-primary-border`、`text-success`、`text-warning`、`text-destructive`.
- 页面布局和自定义组件优先使用Tailwind语义类,例如`bg-background`、`bg-card`、`bg-popover`、`bg-muted`、`text-foreground`、`text-muted-foreground`、`text-primary`、`border-border`、`ring-ring`. 写CSS时使用`hsl(var(--primary))`、`hsl(var(--foreground))`这类框架变量形式.
- Ant Design Vue组件本身应优先通过组件语义属性和框架统一的`theme.token`生效,例如`Button type="primary"`、`Button danger`、`Tag`语义色. 当前应用未默认开启Ant CSS变量模式,页面自定义CSS不得假设`--ant-color-primary`、`--ant-color-text`等`--ant-*`变量一定存在;只有确认组件库已生成对应CSS变量时才允许使用.
- 应用主色、文本色、边框色、背景色都可能由用户在运行时切换;按钮、图标、悬浮面板、弹窗、表格工具栏、列设置和导出功能必须跟随主题变量变化.
- SVG或图标按钮优先使用`currentColor`,由父元素通过语义类或`hsl(var(--...))`控制颜色;不要给图标单独写固定颜色.
- 使用CSS变量时不得为了兜底写入固定色值,例如不得写`var(--primary, #00c48c)`或`var(--ant-color-primary, #00c48c)`. 如果确需兜底,应兜底到另一个框架语义变量或使用`currentColor`、`inherit`、`transparent`.
- 只有品牌Logo、第三方品牌色、数据可视化固定色板或确有业务含义的特殊状态色可以例外;例外必须在代码附近说明原因,并控制影响范围.

## 前端通用CRUD表格工具栏与列设置规则

- 通用CRUD表格必须提供与菜单页面一致的表格工具能力:刷新、全屏/退出全屏、列设置;这些属于框架级表格工具按钮,不要求来源于后端`@CRUD.Op`;按钮必须真实可见、可点击、可验证,不得使用未注册组件导致渲染为零尺寸占位.
- 通用CRUD表格应提供前端导出能力;导出按钮放在表格右上工具区,与刷新、全屏/退出全屏、列设置按钮同组展示.
- 点击导出按钮时必须弹出字段选择弹窗,列出当前列表所有可导出字段;默认勾选字段为当前表格中可见的字段.
- 导出字段顺序默认跟当前表格列顺序一致;如果用户通过列设置调整过表格列顺序,导出弹窗必须使用调整后的实时列顺序.
- 导出字段弹窗必须支持通过上下箭头调整字段位置;最终Excel列顺序以弹窗中的字段顺序为准. 勾选或取消字段不得改变其它字段的相对顺序.
- 导出只针对当前激活的`ListTable`;多`ListTable`页面中,每个列表的导出字段、字段顺序、查询条件、排序状态、列表接口和导出状态必须相互独立.
- 导出数据时必须使用当前列表的查询条件和排序条件循环调用当前列表接口,不得修改页面当前分页状态;分页参数建议使用`pageSize=2000`,从第一页开始请求,直到后端接口返回空数据或返回数量小于请求分页大小为止.
- 导出的单元格值应优先使用列表展示值,例如选择类字段导出标签文本、boolean启用/禁用字段导出业务文案、日期时间字段按页面格式导出;无法安全转换的复杂值才使用稳定的字符串化兜底.
- 导出过程必须有防重复提交、加载状态和失败提示;导出文件由前端JS生成Excel文件并触发下载.
- 列设置面板必须支持列显示/隐藏、列顺序调整、固定到左侧、固定到右侧和恢复默认;固定按钮不额外提供`X`取消按钮,再次点击已激活的左/右固定按钮即取消固定.
- 列设置必须是实时预览:用户在面板中调整列显示、列顺序、列固定时,表格立即变化;保存按钮只负责把当前配置持久化;取消按钮必须恢复到打开面板前的表格状态.
- 恢复默认按钮只先把表格实时恢复为默认预览;只有用户点击保存后,才清除或覆盖持久化配置.
- 列顺序调整必须同时支持拖动排序和上下箭头移动;面板内必须显示拖动排序提示和拖动手柄.
- 固定列时必须自动重排列顺序:左固定列移动到左侧固定组,右固定列移动到右侧固定组,普通列保持在中间;不得因为固定中间列导致表格中间出现空白或错位.
- 列设置持久化配置至少应包含列显示/隐藏、列顺序和列固定状态;多`ListTable`页面必须按具体`ListTable`独立存储.
- 表格数据排序是独立于列顺序调整的功能,不得用列设置里的“列顺序调整”替代表头数据排序.
- 通用CRUD表格必须支持表头数据排序;点击可排序列的表头后,必须立即按当前排序条件重新加载列表数据.
- 表格数据排序状态必须按具体`ListTable`独立维护,并在分页、查询、刷新时继续生效;切换Tab时不得复用其它`ListTable`的排序状态.
- 数据排序请求参数默认使用`orderBy`和`orderDir`;`orderDir`使用后端约定值`Asc`/`Desc`. 如果具体接口有特殊排序参数,必须在页面配置或api服务层显式声明.
- AI辅助实现或维护表头数据排序时,必须把排序状态纳入当前`ListTable`运行时状态,并在表格分页、查询、刷新、Tab切换后保持一致;Ant Table排序事件需兼容单列`sorter`对象和多列`sorter`数组.
- 表头数据排序默认以后端可映射字段为可排序候选,但应允许字段配置显式关闭排序;操作列、虚拟列、复杂展示列和无法被后端字段映射的列不得生成排序请求.
- AI辅助开发通用CRUD页面时,对于启用/禁用语义且字段类型为`boolean`的字段,应实现列表快捷开关能力,并接入页面已有的编辑、修改或更新权限控制;非`boolean`字段不得实现此类快捷开关.
- AI辅助实现启用/禁用快捷修改时,必须调用`api`目录下的服务接口,不得绕过页面接口规则;请求载荷只应包含主键、乐观锁字段和当前boolean状态字段,不得把整行数据直接提交.
- 启用/禁用快捷修改实现必须包含请求中防重复提交、失败恢复原值和错误提示;成功后保持当前分页、查询和排序状态重新加载数据.

## 前端公共基础设施使用规则

- 事件总线、顶部栏扩展区和可拖拽浮动面板属于前端公共基础设施，业务模块必须通过 `@levin/admin-framework/framework-commons` 使用，不得直接引用 `@vben/layouts`、`packages/effects/layouts`、`shared/draggable-floating-panel-*` 或其它框架内部实现文件.
- 事件总线只负责在同一前端运行时内发送事件、添加监听器、移除监听器、启用/禁用监听器和查看当前监听器；它不限制事件分类，不替代 API service、Pinia、路由、组件 props 或持久化存储.
- 发送事件必须使用固定三元组：事件类型 `type`、事件主题 `topic`、事件数据 `data`. 监听器可以按 `type` 和 `topicPattern` 监听，`topicPattern` 支持 `*` 和 `?` 通配符.
- 业务模块添加事件监听器时必须填写备注 `remark`，说明监听器用途；组件或功能卸载时必须清理监听器. 临时停用监听器应使用启用/禁用能力，不应为了排查问题删除业务注册逻辑.
- 跨模块通知或框架能力监听 API 返回结果时，应使用事件总线，不允许业务模块之间直接互相调用内部实现. API 请求事件规则继续遵循 `前端API开发规则.md`.
- 顶部栏固定控件必须使用 `useLayoutHeaderExtensionArea('center')` 或 `useLayoutHeaderExtensionArea('right')` 注册；不得用 `position: fixed`、页面浮动面板或直接修改框架内部顶部栏模板来模拟.
- 顶部栏扩展区的承载层必须占满顶部条高度；业务控件本身可以按自身交互语义设置按钮、选择器或标签尺寸，不应通过缩短承载层高度来控制控件大小.
- 顶部栏扩展项必须提供稳定 `id`，并通过 `order` 控制顺序. 同一区域重复 `id` 会替换旧项；通过组合式函数 `add` 创建的项会在当前组件卸载时自动清理.
- 需要在整个浏览器窗口内浮动、可拖拽、可记忆位置的控件，应使用框架根应用已挂载的全局 `DraggableFloatingPanelHost`，业务模块只通过 `useDraggableFloatingPanels(scope)` 或 `addDraggableFloatingPanel` 注册面板. 多个浮动面板必须使用稳定 `scope`、稳定 `id`，需要持久化位置时必须提供独立 `storageKey`.
- 固定在顶部栏的控件和浏览器窗口级浮动面板是两个不同场景：顶部栏控件用顶部栏扩展区，窗口级悬浮控件用可拖拽浮动面板，不得混用.
- 公共基础设施的详细使用说明统一维护在 `docs/frontend-common-infrastructure.md`; 事件总线的 API 请求事件细节同时参考 `docs/framework-event-bus.md`.

## 前端窗口级悬浮面板规则

- 本节是前端公共基础设施规则中“顶部栏扩展区”和“可拖拽浮动面板”的窗口级细化要求；新增业务模块应同时参考 `docs/frontend-common-infrastructure.md`.
- 页面级上下文切换器、全局筛选条、当前商户/当前项目等位置型控件，应按交互语义选择独立组件，不应在各业务页面重复实现定位逻辑.
- 可拖拽悬浮面板是浏览器窗口级公共能力，不占用页面正常布局空间，默认定位和拖拽边界以根文档视口为准，而不是内容区域、表格区域或 iframe 内部区域.
- 可拖拽悬浮面板默认显示在根文档视口顶部居中；同一 host 内多个面板按 `placementGap` 做位移，避免完全重叠.
- 框架根应用负责挂载全局 `DraggableFloatingPanelHost`; 业务页面不得为了让面板跨出内容区而自行在内容区包一层 host. 如果业务模块运行在 iframe 内，iframe 内组件不能直接跨出 iframe 边界，应由宿主根应用注册面板或通过宿主提供的桥接能力注册.
- 可拖拽悬浮面板只能通过框架提供的左上角拖动图标移动，面板内容区不得整体绑定拖拽事件，避免影响业务组件内部的点击、选择、滑动、拖放等交互. 面板默认支持点击该图标收缩到只剩图标，再次点击展开；拖动时不得误触发展开或收缩.
- 可拖拽悬浮面板默认不限制最大宽高；需要限制尺寸时，由业务内容组件或 `panelClass` 自行设置 `max-width`、`max-height`、滚动区域等样式.
- 同一页面存在多个可拖拽悬浮面板时，应通过 `useDraggableFloatingPanels(scope)` 的 `add`/`remove` 方法新增或删除面板；每个面板必须使用稳定的 `id` 或独立的 `storageKey`.
- 业务模块注册可拖拽悬浮面板时应填写可读 `name`，用于后续管理界面、调试、排查和拖动手柄提示；`id` 只作为程序唯一标识，不应让用户或运维人员依赖技术 id 识别面板用途. 拖动手柄必须使用项目已有 Tooltip 组件，不使用浏览器原生 `title`; 提示必须包含面板名称，延迟 1.5 秒显示，按下手柄后必须立即关闭提示，展开态提示“点击可收缩，按住可拖动”，收缩态提示“点击可展开，按住可拖动”.
- 页面级浮动面板应优先通过 `useDraggableFloatingPanels(scope).add` 注册，当前页面组件卸载时会自动移除；入口应用或宿主应用需要常驻的浮动面板才使用全局 `addDraggableFloatingPanel`，并由调用方保存清理函数或主动移除.
- 需要记住用户拖动位置时，应为 `DraggableFloatingPanel` 提供稳定的 `storageKey`; 不同页面、不同业务上下文的悬浮面板不得共用同一个存储键.
- 固定在顶部栏的控件不是悬浮面板，不得使用 `position: fixed` 或可拖拽面板模拟；它们必须注册到现有顶部栏组件提供的 `center` 或 `right` 区域中，跟随顶部栏高度、布局和显示隐藏策略.
- 固定顶部中央内容使用 `useLayoutHeaderExtensionArea('center')` 注册；固定顶部右侧内容使用 `useLayoutHeaderExtensionArea('right')` 注册。业务卸载时应清理注册项，组合式函数会自动清理本组件内通过 `add` 创建的项.
- 业务模块只能从 `@levin/admin-framework/framework-commons` 引入顶部扩展区和浮动面板服务，不得直接引用 `@vben/layouts`、`packages/effects/layouts` 或 `shared/draggable-floating-panel-*` 内部文件；框架内部文件只承载实现，公共导出边界以 `framework-commons` 为准.

业务模块调用方式:

```vue
<script setup lang="ts">
import { useDraggableFloatingPanels } from '@levin/admin-framework/framework-commons';
import CurrentMerchantSelect from './current-merchant-select.vue';

const floatingPanels = useDraggableFloatingPanels('project-publish');

floatingPanels.add({
  id: 'merchant-filter',
  component: CurrentMerchantSelect,
  panelProps: {
    panelClass: 'px-5 py-3',
    storageKey: 'project-system:publish:merchant-filter',
  },
});
</script>
```

固定到当前顶部栏中央:

```vue
<script setup lang="ts">
import { useLayoutHeaderExtensionArea } from '@levin/admin-framework/framework-commons';
import CurrentMerchantSelect from './current-merchant-select.vue';

const topCenter = useLayoutHeaderExtensionArea('center');

topCenter.add({
  id: 'project-publish-merchant',
  component: CurrentMerchantSelect,
  class: 'px-2',
});
</script>
```

固定到当前顶部栏右侧:

```vue
<script setup lang="ts">
import { useLayoutHeaderExtensionArea } from '@levin/admin-framework/framework-commons';
import CurrentProjectSelect from './current-project-select.vue';

const topRight = useLayoutHeaderExtensionArea('right');

topRight.add({
  id: 'project-publish-project',
  component: CurrentProjectSelect,
  class: 'mr-2',
});
</script>
```

如果业务本身直接拥有布局模板，也可以使用现有顶部栏插槽 `#header-top-center` 和 `#header-top-right`；普通页面组件不能直接改顶部栏模板时，使用 `useLayoutHeaderExtensionArea` 注册.

## 管理后台页面开发总原则

前端项目页面的开发要基于后端控制器类和注解元数据，不是只基于 OpenAPI 文档，也不是前端自行猜测页面结构。

规则、源码、运行时信息发生不一致时，优先级如下：

1. 当前后端源码中的注解定义，例如 `com.levin.commons.ui.annotation`、`com.levin.commons.rbac.ResAuthorize`。

2. 当前业务模块中的 Spring MVC 控制器源码及其父类源码。

3. 当前运行环境实际暴露的接口信息，例如 OpenAPI 或运行时调试结果，仅用于辅助确认是否暴露。

4. 前端规则文档和页面实现。

5. 旧版 jar、旧版生成代码、旧版 OpenAPI、历史页面实现只能作为排查线索，不能覆盖当前源码。

API 组织与复用规则独立维护在 `前端API开发规则.md`。

页面字段、表单、列表、选项、布局规则独立维护在 `前端页面开发规则.md`。

页面操作、按钮、`CRUD.Action`、显隐和权限规则独立维护在 `前端页面操作规则.md`。

图片和文件上传规则独立维护在 `前端页面上传规则.md`。

表单校验规则独立维护在 `前端页面校验规则.md`。

前端开发服务器、Nginx、后端回退和资源代理规则独立维护在 `前端部署代理规则.md`。

所有前端页面开发规则，都是用来指导页面开发的，不是用来构建页面代码生成器的。

不允许把这些规则实现成页面代码生成器、运行时元数据生成器或通用页面自动拼装系统。

不允许用控制器元数据、OpenAPI 元数据或其他中间结构替代真实页面开发。

规则的作用是指导开发者如何分析后端源码、如何设计页面结构、如何确定字段、按钮、权限、布局和组件，而不是生成页面代码本身。

每个菜单页面仍应有明确的独立页面目录和页面实现。

每个菜单对应一个明确的前端页面目录。

每个菜单页面应有自己的页面入口文件，例如 `index.vue`。

页面之间可以复用公共组件，但不能把多个菜单页面混成一个“通用生成页”。

共享组件只负责复用局部能力，不负责替代页面本身。

## 管理后台页面开发补充规则

### 一、页面开发主依据

1. 在本项目中，只要存在对应的 Spring MVC 控制器源码，就不得使用 OpenAPI 作为页面开发主依据。

2. OpenAPI 只能作为辅助信息，用于以下场景：
   - 辅助确认某个控制器是否在当前运行环境实际暴露。
   - 辅助确认某个控制器方法是否实际暴露。
   - 辅助排查接口路径、HTTP 方法、运行时条件注解是否生效。
   - 辅助排查源码与运行时暴露不一致的原因。

3. OpenAPI 不能用于决定以下内容：
   - 页面是否存在。
   - 页面字段。
   - 页面按钮。
   - 菜单归属。
   - 权限点。
   - 显隐条件。
   - 表单布局。
   - 组件类型。
   - 业务交互行为。

4. 页面开发的主依据必须是后端源码：
   - 业务控制器。
   - 父类控制器。
   - 控制器方法参数。
   - 控制器方法返回值。
   - 方法参数对应的 DTO / BO / VO / 基本类型参数。
   - 返回对象对应的 DTO / Info / VO / BO。
   - 实体类及其关联类型。
   - 相关注解，如 `@CRUD`、`@CRUD.ListTable`、`@CRUD.Op`、`@DisableApiOperation`、`@Operation`、`@Tag`、`@Schema`、`@Options`、Java Validation 注解等。

### 二、有效控制器判定

1. 只有有效的 Spring MVC 控制器才对应前端页面模块。

2. 一个业务控制器要被视为有效页面控制器，必须同时满足：
   - 是 Spring Bean。
   - 不是抽象类。
   - 是 Spring MVC 控制器，例如存在 `@RestController`、`@Controller`，或通过其他 Spring 配置明确注册为 MVC 控制器。
   - 自身或父类存在 `@CRUD` 注解。

3. 如果业务控制器自身没有 `@CRUD`，但父类控制器有 `@CRUD`，则可以继承父类的 CRUD 页面定义。

4. 如果业务控制器是抽象类，即使源码里有 `@RequestMapping`、`@CRUD` 或继承了 CRUD 控制器，也不能直接作为前端页面控制器。

5. 如果业务控制器没有作为 Spring Bean 注册，例如缺少控制器注解、没有被组件扫描、被条件注解禁用，不能直接作为页面控制器。

6. `@ConditionalOnProperty` 等条件注解需要纳入判断。
   - 如果当前配置明确禁用了该控制器，则对应页面不应展示。
   - 如果无法仅凭源码判断条件是否生效，可以结合运行时信息或 OpenAPI 辅助确认。

7. 如果不确定控制器是否有效，可以结合 OpenAPI 辅助判断。
   - OpenAPI 中存在对应接口，说明当前运行环境大概率暴露了该控制器或方法。
   - OpenAPI 中不存在对应接口，应继续检查是否由 `@DisableApiOperation`、条件注解、抽象类、非 Spring Bean、运行环境配置、扫描范围等导致。

8. `@DisableApiOperation` 的判断优先级高于 OpenAPI。
   - 即使某方法在源码中存在，只要被 `@DisableApiOperation` 禁用，前端不得显示为页面动作。
   - 如果 OpenAPI 暴露与 `@DisableApiOperation` 不一致，应记录不一致并进一步确认原因。

9. 前端页面开发前，应先整理有效控制器清单：
   - 控制器类名。
   - 是否 Spring Bean。
   - 是否抽象类。
   - 是否有 `@CRUD` 或继承 `@CRUD`。
   - 是否被条件注解启用。
   - 对应基础路径。
   - 对应菜单/路由路径。
   - OpenAPI 是否能辅助确认当前运行时暴露。

### 三、控制器优先级

1. 业务控制器优先级高于父类控制器。

2. 如果业务控制器没有重写某个父类方法：
   - 使用父类方法上的 Spring MVC 映射。
   - 使用父类方法上的 `@Operation`。
   - 使用父类方法上的 `@CRUD.ListTable`。
   - 使用父类方法上的 `@CRUD.Op`。
   - 使用父类方法参数和返回值作为页面字段来源。

3. 如果业务控制器重写了父类方法：
   - 业务控制器方法优先。
   - 父类方法作为缺省元数据来源。
   - 不能因为业务控制器方法未重复声明某些注解，就认为这些注解不存在。

### 四、方法级注解继承与覆盖

1. `@CRUD.ListTable` 和 `@CRUD.Op` 是方法级页面元数据，可以从父类控制器方法继承。

2. 判断某个控制器方法的前端页面元数据时，应同时检查：
   - 当前业务控制器方法。
   - 父类控制器同签名或同语义方法。
   - 被重写方法上的 Spring MVC 映射注解。
   - 被重写方法上的 `@Operation`。
   - 被重写方法上的 `@CRUD.ListTable`。
   - 被重写方法上的 `@CRUD.Op`。

3. 如果业务控制器没有重写某个父类方法：
   - 直接使用父类方法上的 Spring MVC 映射和 CRUD 注解。
   - 例如父类 `list()` 上有 `@CRUD.ListTable`，业务控制器未重写，则该列表页仍然有效。

4. 如果业务控制器重写了父类方法，但没有重新声明部分注解：
   - 应继承父类同方法上的缺失注解。
   - 例如业务控制器重写 `update()`，只声明了 `@CRUD.Op(visibleOn = "...")`，但没有重新声明 `@PutMapping` 或 `@Operation`，前端仍应继承父类 `update()` 的请求路径、HTTP 方法和操作名称。
   - 业务控制器只覆盖它显式声明的注解属性，其余未声明属性继续从父类方法继承。

5. 如果业务控制器重写方法并重新声明了同类注解：
   - 当前业务控制器方法优先。
   - 父类方法只作为缺省值来源。
   - 子类显式声明的 `@CRUD.Op` 属性覆盖父类 `@CRUD.Op` 对应属性。
   - 子类显式声明的 `@CRUD.ListTable` 属性覆盖父类 `@CRUD.ListTable` 对应属性。
   - 子类显式声明的 `@Operation`、`@RequestMapping` / `@GetMapping` / `@PostMapping` 等也覆盖父类对应信息。

6. 继承和覆盖应按“属性级合并”，不要简单地用子类注解整体替换父类注解。
   - 父类：`@CRUD.Op(confirmText = "None")`
   - 子类：`@CRUD.Op(visibleOn = "status == 'Draft'")`
   - 合并后应同时拥有：
     - `confirmText = "None"`
     - `visibleOn = "status == 'Draft'"`
   - 除非子类显式设置了同名属性，否则父类属性应保留。

7. 方法匹配应优先使用 Java 重写关系或方法签名。
   - 同名方法只是辅助判断。
   - 如果存在重载方法，应按参数类型、HTTP 映射和语义进一步区分。
   - 不应只按方法名粗暴合并所有注解。

8. `@DisableApiOperation` 仍然具有最终禁用优先级。
   - 即使方法从父类继承了 `@CRUD.Op` 或 `@CRUD.ListTable`，只要当前有效控制器被 `@DisableApiOperation` 禁用该方法或路径，前端不得展示对应页面或按钮。

9. 如果继承关系无法通过源码静态判断完整确认，可以结合 OpenAPI 辅助判断方法是否实际暴露；但页面元数据仍以控制器源码和注解继承/覆盖规则为主。

### 五、控制器方法参数决定页面字段

1. 页面字段必须通过控制器方法签名和源码分析确定。

2. 查询表单字段来自列表查询方法的参数，而不是固定命名的 `QueryXxxReq`。

3. 新增表单字段来自新增方法的参数，而不是固定命名的 `CreateXxxReq`。

4. 编辑表单字段来自编辑方法的参数，而不是固定命名的 `UpdateXxxReq` 或 `SimpleUpdateXxxReq`。

5. 删除、批量、业务操作表单字段来自对应操作方法的参数。

6. 如果方法参数是 DTO / BO / VO，则读取该类型字段及其注解。

7. 如果方法参数是基本类型、枚举、集合、`@RequestParam`、`@PathVariable` 等，则按参数本身和参数注解生成对应输入项。

8. 如果方法有多个参数，应按 Spring MVC 参数绑定语义区分：
   - 请求体参数。
   - 查询参数。
   - 路径参数。
   - 表单参数。
   - 分页参数。
   - 当前请求上下文参数。
   - 注入上下文参数。

9. 上下文注入类参数、分页参数、请求上下文参数等不应错误展示为业务表单字段。

10. 列表字段来自列表方法返回值中的数据项类型，或控制器/注解明确指定的列表实体/结果类型。

### 六、页面与按钮追溯要求

1. 每个菜单页面必须能明确追溯到：
   - 对应哪个控制器。
   - 对应哪个列表方法。
   - 对应哪些按钮方法。
   - 对应哪些禁用规则。
   - 对应哪些参数类型。
   - 对应哪些返回类型。

2. 每个页面至少要明确：
   - 列表来自哪个 `@CRUD.ListTable` 方法。
   - 页面级按钮来自哪些 `opRefTargetType=None` 方法。
   - 单条记录按钮来自哪些单条 `@CRUD.Op` 方法。
   - 批量按钮来自哪些批量 `@CRUD.Op` 方法。
   - 被禁用的方法为什么不展示。

### 七、字段规则应用顺序

1. 先根据控制器方法参数、返回值和源码注解确定字段来源。

2. 再按以下规则执行字段过滤、排序、组件、选项、上传、校验：
   - 《前端页面开发规则》
   - 《前端页面上传规则》
   - 《前端页面校验规则》

3. 如果控制器源码和 OpenAPI 展示不一致，以控制器源码为准。

4. 如果源码信息不足，应优先补充后端注解或与用户确认，不应通过 OpenAPI 猜测前端行为来绕过源码缺失。

### 八、页面操作规则

页面、按钮、`CRUD.Action`、显隐条件、确认框、操作容器和权限匹配规则，统一遵守 `前端页面操作规则.md`。

总规则中不重复维护这些细节，避免页面操作规则在多个文件中出现不一致。

重大需求、跨页面规则、公共组件边界、路由策略、权限策略、接口组织方式、视觉体系等涉及面较广的调整，应先更新 `openspec/` 中对应需求或与用户确认方案，再开始编码。局部页面修复、按既定规则补字段、修布局或修接口调用等低风险实现问题，可以直接执行并验证。

前端开发流程默认采用结构化的需求拆解、计划执行、验证收口流程；不要跳过需求整理直接改页面。

前端项目统一使用语言：简体中文。页面文案、交互提示、占位文案、错误提示、注释、规则文档、说明文字、测试描述、开发输出等，默认都必须使用简体中文；仅在第三方字段名、标准协议名、框架保留关键字等不可翻译场景下保留原文。

涉及注解语义、按钮行为、继承覆盖或复杂状态流转时，可优先阅读以下后端代码作为参考样例：

- `com.levin.commons.ui.annotation` 包
- `admin-api/src/main/java/com/levin/oak/base/controller/BizArticleController.java`
- 对应的父类控制器，例如 `admin-api/src/main/java/com/levin/oak/base/controller/base/article/ArticleController.java`

### 九、页面目录归组规则

1. 前端页面目录应按当前项目的模块包名归组存放，以便后续支持多个模块并存。

2. 模块目录名不直接按 Java 包路径展开，而应使用扁平化后的模块包名。包名中的 `.` 统一替换为 `_`。例如：`com.levin.oak.base` 应对应为 `com_levin_oak_base`。

3. 页面目录结构不应过深。不使用 `com/levin/oak/base/...` 这种多层目录展开方式。应在页面根目录下使用一层模块目录承载该模块下的页面。

4. 页面目录组织顺序应为：先按模块包名目录归组。再在模块目录下按具体页面资源建立独立页面目录。

5. 每个页面仍应保持自己的独立目录和页面实现文件。页面入口文件、`config.ts`、页面子组件、页面专属弹窗、页面专属工具函数等，都应归属在对应模块目录下。

6. 公共组件可以继续放在共享目录，但业务页面及其专属实现必须保留模块归属信息。

7. 这种模块包名归组方式应同时避免：不同模块同名页面冲突。路由命名冲突。页面实现长期平铺在同一层目录下导致维护混乱。

8. 后续新增模块时，应优先在新的模块包名目录下新增页面，而不是继续平铺到现有页面目录中。
