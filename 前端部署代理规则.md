# 前端部署代理规则

本文件用于约束管理后台前端开发服务器、生产 Nginx、静态资源和后端动态路径的代理关系。

## 一、前端未命中转后端

1. 前端 Nginx 配置中，当请求未命中前端静态资源或前端页面路由时，不应直接返回前端 404 页面。

2. 对于前端未命中的请求，应优先转发到后端接口，由后端继续处理。

3. 该规则适用于以下场景：
   - 后端控制器直接提供页面内容。
   - 后端提供 AMIS 页面或 HTML 页面。
   - 后端提供模块级动态路由页面。
   - 某些路径虽然前端本地没有页面文件，但实际上由后端负责响应。

4. 前端 Nginx 或开发代理应优先区分三类请求：
   - 前端静态资源请求。
   - 前端本地页面路由请求。
   - 前端未命中但可能属于后端处理范围的请求。

5. 当请求属于前端未命中场景时，应先尝试转发给后端，而不是立即显示“未找到页面”。

6. 只有在前端未命中、且后端也无法处理该请求时，才应最终返回 404。

7. 这条规则适用于开发环境代理和生产环境 Nginx 配置。
   开发环境的前端代理策略应尽量与生产环境保持一致。

8. 前端页面开发时，不应依赖“前端找不到就报错”来判断页面是否存在。
   需要同时考虑该路径是否本应由后端处理。

## 二、后端资源和模块路径代理

1. 后端资源路径必须被前端开发服务器和 Nginx 正确代理，否则上传后的图片、文件和后端直出内容会无法回显。

2. 代理路径应覆盖：
   - 文件资源路径，例如 `/lfs`。
   - 后端管理路径，例如 `/admin`。
   - API 文档路径，例如 `/api-docs`、`/swagger`。
   - 证书校验路径，例如 `/.well-known/acme-challenge`。
   - 各后端模块 API 前缀，例如当前模块 `/com.levin.oak.base`。

3. `/com.levin.oak.base` 只是一个后端模块前缀，不代表所有模块。
   后续新增后端模块时，应按模块前缀补充代理，而不是把某个模块路径写死成全局唯一 API 根路径。

4. `VITE_GLOB_API_URL` 可以表示当前模块的默认 API 地址，但项目整体仍以 `/` 为根地址。
   通用配置、公共组件和跨模块 API 层不得假设所有接口都位于同一个模块前缀下。

5. 前端路由、后端页面回退链、AMIS/HTML 页面加载、上传资源回显，都应共享同一套代理策略，避免开发环境和生产环境行为不一致。
